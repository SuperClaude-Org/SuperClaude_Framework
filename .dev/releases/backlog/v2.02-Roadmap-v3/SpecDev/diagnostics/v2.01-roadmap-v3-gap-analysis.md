## 1. EXECUTIVE SUMMARY

The v2.01-Roadmap-v3 diagnostic workflow produced **22 planning artifacts** across 4 phases (5 root-cause investigations, 5 solution proposals, 5 adversarial debates, 4 checkpoints) plus 3 post-spec review documents (reflection-final.md, spec-panel-review.md, DVL-BRAINSTORM.md). From that body of work, the final sprint-spec.md constitutes roughly **35% of all contemplated ideas**, with the remainder deferred, rejected, or silently dropped.

| Category | Count | Notes |
|----------|-------|-------|
| Ideas that made the final spec | ~42 | Includes refined/transformed versions |
| Ideas explicitly deferred to future sprint | ~18 | RC3+S03, RC5+S05, DVL scripts |
| Ideas explicitly rejected (with rationale) | ~11 | Options A, C, D in key decisions; ranking ceremony cuts |
| Ideas silently dropped (no disposition) | ~19 | Fell through without formal rejection or deferral |
| Failure modes identified but unaddressed | 7 | Timeouts, YAML parse errors, recursion depth, etc. |
| Methodology innovations | 6 | Reusable diagnostic patterns |

The most significant gap: the planning process contemplated a rich **DVL (Deterministic Verification Layer)** of 10 verification scripts, 6 anti-hallucination strategies, a 3-tier orchestration architecture, and a sentinel-file audit trail — all of which are referenced in the spec only as "Future Work" with 3 scripts named. Separately, two full solution pairs (RC3+S03 and RC5+S05) received equivalent analytical depth to the three that made the spec, but were cut on combined-score grounds.

## 2. IDEAS THAT MADE IT INTO THE FINAL SPEC

For each major spec element, this section traces its origin and transformation.

### 2.1 Task 0.0: Skill Tool Probe

**What the spec says**: Dispatch a single Task agent to empirically test whether the Skill tool works cross-skill before all implementation begins. Four decision-gate outcomes mapped to sprint adaptations.

**Origin**: Converged from four independent sources:
- **spec-panel-review.md** — all four panelists (Brooks, Lamport, Leveson, Weinberg) unanimously recommended this as the single highest-value pre-implementation action
- **reflection-final.md** (IMP-01) — self-review agent proposed "Task 0.1" with near-identical method and decision-gate structure
- **debate-01** (Unresolved Concern 1) — debate produced Risk R1 at probability 0.40, explicitly noting the primary path had never been tested
- **CP-P3-END.md** — Phase 3 checkpoint carried UC-01 forward as the top unresolved concern

**Transformation**: The probe was originally proposed as a "Task 0.1" in reflection-final.md. The panel review reframed it as "Task 0.0" and specified it must precede everything, not follow the allowed-tools changes (Tasks 1.1/1.2). The sprint-spec.md adopted the panel's framing including the "already running" semantics analysis question and the 4-outcome decision tree.

**What was added from spec vs. reflection**: The spec added an explicit question about the semantics of "Do not invoke a skill that is already running" — does the constraint apply to (a) the exact same skill, (b) any skill while another is active, or (c) the same skill instance? This nuance came from Leveson's hazard analysis (H1) in the panel review.

### 2.2 Epic 1: Invocation Wiring (RC1 + S01)

#### Skill tool addition to allowed-tools (Tasks 1.1, 1.2)
**Origin**: RC1 root-cause analysis. The evidence was concrete: grep of roadmap.md and SKILL.md frontmatter showed `Skill` absent from `allowed-tools`. Ranked-root-causes.md Fix 1 specified the exact change.

**No transformation**: This was the most literal translation from analysis to spec. The change was always "add `Skill` to two allowed-tools lines."

#### Task Agent Wrapper as primary invocation path (Task 1.3)
**Origin**: debate-01 (invocation-wiring), Option B. The debate evaluated four options:
- Option A: Direct Skill call (no Task agent)
- **Option B: Task agent dispatches Skill call** — selected
- Option C: Agent Loader Template (new infrastructure)
- Option D: File-based handoff with separate orchestration

**Transformation**: Option B won the debate by scoring highest on blast radius (low) and feasibility. The reflection-final.md (IMP-02) reconsidered this, noting Option A (direct Skill call from main agent) eliminates the R1 risk entirely and should be considered. The final spec kept Option B as primary while the Task 0.0 probe determines whether to adapt.

#### 5-step fallback protocol F1-F5 (Task 1.4)
**Origin**: Multiple sources converged:
- solution-05-claude-behavior.md (S05) proposed a probe-and-branch strategy; its "inline pipeline" became the fallback basis
- debate-01 produced the requirement that fallback must cover all 3 error types
- spec-panel-review.md (Lamport, Change 5) specified the fallback as a formal state machine with defined inputs, outputs, and failure actions per step — this produced the F1-F5 table structure

**Transformation**: The fallback went through substantial elaboration between debate phase and spec. The debates established "fallback is necessary" and "cover three error types." The panel (Lamport) specified the state-machine format. The final spec merged both into the full F1-F5 table with failure_stage values for each step. The `fallback_mode: true` field was added at Lamport's recommendation (subsequently included in Task 3.1's 9-field schema, reconciling an inconsistency the panel identified).

## 8. SCORING AND RANKING DECISIONS

The diagnostic workflow used three scoring formulas that shaped the sprint's scope.

### 8.1 Problem Scoring Formula

```
problem_score = (likelihood * 0.6) + (impact * 0.4)
```

Weights: likelihood (0.6) > impact (0.4). Rationale: a lower-likelihood high-impact problem has fewer expected-value consequences than a high-likelihood moderate-impact problem. This weighting favored RC1 (likelihood 0.90, impact 0.90) and RC5 (likelihood 0.85, impact 0.70) over RC3 (likelihood 0.70, impact 0.75).

**Key scoring decision**: RC3 had its likelihood score cut from 0.95 to 0.70 during cross-validation (the most aggressive reduction across all five). This demotion moved RC3 from Rank 2 to Rank 5 in problem scoring, effectively removing it from sprint scope. If the self-reported 0.95 had been accepted, RC3 would have scored 0.870 and ranked second — it would have been in the sprint.

### 8.2 Solution Scoring Formula

```
solution_score = (fix_likelihood * 0.5) + (feasibility * 0.3) + (low_blast_radius * 0.2)
```

Weights: fix_likelihood (0.5) dominates. Feasibility and blast radius are secondary. This formula paradoxically ranked S04 (Return Contract, 0.806) above S01 (Invocation Wiring, 0.775) despite S01 being the problem-score leader. S04 earned high feasibility (0.83) and low blast radius (0.85), while S01's fix_likelihood (0.760) was penalized by uncertainty about Skill tool cross-invocation.

**Key consequence**: The formula produced a top-3 where the return contract ranked #2 even though it addresses a latent defect (RC4). This was acceptable because RC4 would surface immediately after the primary fixes — but it's notable that the formula can surface latent-defect solutions above primary-fix solutions.

### 8.3 Combined Ranking Formula

```
combined = (problem_score * 0.5) + (solution_score * 0.5)
```

Equal weighting between problem severity and solution quality. Top 3 results:

| Rank | Pair | Combined |
|------|------|---------|
| 1 | RC1+S01 | 0.838 |
| 2 | RC4+S04 | 0.778 |
| 3 | RC2+S02 | 0.776 |

**The margin is extremely small**: RC4+S04 and RC2+S02 differ by only 0.002. The selection rationale in the spec acknowledges this, attributing RC4's edge to "solution score strength (highest feasibility in the set)." If the problem scores had been used alone, RC2 (0.770) would have outranked RC4 (0.750) and the order would have flipped.

**The cutoff**: RC5+S05 scored 0.761 and RC3+S03 scored 0.728. These fell below the top-3 threshold. RC5+S05 missed the cut by 0.017 from RC2+S02 — a very small margin. A 5% change in any of RC5's or S05's component scores would have put it in the sprint.

### 8.4 What the Formulas Got Right and Wrong

**Right**: The formulas enforced a discipline that prevented scope creep. Without explicit cutoffs, all five pairs might have been included, producing an undeliverable sprint. The formulas made the RC3/RC5 deferral appear objective rather than arbitrary.

**Wrong** (as the panel noted): The precision is false. Inputting 0.90 vs. 0.85 for likelihood is not an empirically derivable distinction — it is an expert estimate. The formula processes this apparent precision as if it were measurement, not estimation. Brooks recommended replacing the formulas with a simple ordered list; Weinberg suggested qualitative assessments in the risk register. Both recommendations were rejected in the final spec.
