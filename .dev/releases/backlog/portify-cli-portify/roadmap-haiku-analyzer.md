# Project Roadmap: `cli-portify` Portification Pipeline

## 1. Executive summary

`cli-portify` is an enterprise-complexity workflow-to-CLI portification pipeline that converts an inference-driven SuperClaude skill into a deterministic, resumable Python CLI runner. The extraction indicates:

- **65 total requirements**
  - **47 functional**
  - **18 non-functional**
- **Complexity**: `0.92` / **enterprise**
- **Domains**: `8`
- **Risks**: `12`
- **Dependencies**: `11`
- **Success criteria**: `14`
- **Extraction mode**: `full`

### Analyzer assessment

This is not a simple CLI feature. It is a **multi-phase orchestration system** with:

1. strict prerequisite validation,
2. sequential pipeline execution,
3. Claude-assisted artifact generation,
4. enforced gate validation,
5. user review checkpoints,
6. release-spec synthesis,
7. a bounded panel-review convergence loop,
8. resumability and diagnostics,
9. final contract emission.

### Primary delivery objective

Deliver a new `src/superclaude/cli/cli_portify/` module and register it in `src/superclaude/cli/main.py` so that users can run a supervised, resumable portification workflow with reliable outputs and clear failure handling.

### Analyzer priorities

1. **Establish correctness boundaries early**:
   - path resolution,
   - name collision detection,
   - workdir isolation,
   - artifact validation,
   - gate behavior.
2. **Reduce failure amplification** in Claude-assisted steps:
   - timeout handling,
   - output monitoring,
   - retry logic,
   - clear diagnostics.
3. **Resolve open questions before deep implementation** where they affect contracts:
   - turn accounting,
   - phase contract schema,
   - hash algorithm,
   - failure type taxonomy,
   - debug mode semantics.
4. **Prove resumability and gate enforcement** before polishing TUI/logging.

## 3. Risk assessment and mitigation strategies

## Highest-priority risks

### 1. Claude subprocess stall
- **Severity**: High
- **Impact**: Pipeline hang, resource waste, no final artifact
- **Mitigation**
  1. Enforce hard timeouts per step.
  2. Implement stall detection using output growth rate.
  3. Define kill policy explicitly:
     - preferred: `SIGTERM` grace period, then `SIGKILL`
  4. Emit diagnostic context on timeout/stall.
- **Roadmap response**
  - Address in Phases 2 and 8, before relying on long-running synthesis/review steps.

### 2. Turn budget exhaustion before Steps 10–11
- **Severity**: High
- **Impact**: Late-stage halt after substantial cost
- **Mitigation**
  1. Pre-step ledger checks.
  2. Budget forecast warnings before expensive phases.
  3. Accurate `suggested_resume_budget`.
  4. Resume command emission in all halted states.
- **Roadmap response**
  - Implement in Phase 2, validate in Phase 10.

### 3. Placeholder leakage into final release spec
- **Severity**: High
- **Impact**: Invalid downstream artifact; false readiness
- **Mitigation**
  1. Explicit placeholder scan.
  2. Gate failure with exact placeholder reporting.
  3. Retry once where appropriate.
- **Roadmap response**
  - Implement in Phases 3 and 6.

### 4. Panel review escalation without downstream-ready output
- **Severity**: High
- **Impact**: Partial output that may be mistaken for completion
- **Mitigation**
  1. Strict state machine.
  2. Separate `status` from `downstream_ready`.
  3. Visible escalation details in `panel-report.md`.
- **Roadmap response**
  - Core requirement of Phase 7.

## Medium-priority risks

### 5. Ambiguous workflow resolution
- **Mitigation**
  - Fail early with candidate list; require explicit path.

### 6. Name collision with non-portified module
- **Mitigation**
  - Marker-based overwrite rule in `__init__.py`.

### 7. Gate false positives from line-count thresholds
- **Mitigation**
  - Keep thresholds configurable or centralized.
  - Ensure prompts explicitly instruct comprehensive output.

### 8. Approval file corruption
- **Mitigation**
  - Prefer YAML parse + schema validation rather than raw string checks.

### 9. Missing `EXIT_RECOMMENDATION`
- **Mitigation**
  - Gate enforcement plus distinct status classification.

### 10. Agent discovery gaps
- **Mitigation**
  - Emit warnings, do not silently suppress.

## Low-priority risks

### 11. Template size handling > 50 KB
- **Mitigation**
  - Resolve mechanism in Phase 0, implement in Phase 6.

### 12. Unbounded log growth
- **Mitigation**
  - Keep logs workdir-scoped; document cleanup expectations.

### B. Behavioral validation
1. Verify pause/resume at both review gates.
2. Verify `--dry-run` stops after Phase 2 boundary.
3. Verify timeout classification uses `124 => TIMEOUT`.
4. Verify gate failures block pipeline when required.
5. Verify convergence loop stops at:
   - success when zero CRITICALs remain
   - escalation after 3 iterations

### C. Operational validation
1. Verify signal interruption produces `INTERRUPTED`.
2. Verify budget exhaustion produces halt with resume guidance.
3. Verify logs and diagnostics are generated on failure.
4. Verify downstream readiness only when `overall >= 7.0`.

Observability should begin earlier than the natural requirement order because:
- long Claude steps are failure-prone,
- diagnostics are needed before synthesis/review debugging,
- resumability is difficult to trust without execution traces.

## Final analyzer recommendations

1. **Do not begin full implementation until Phase 0 open questions are resolved** for turn accounting, hashing, failure typing, and debug semantics.
2. **Treat gates and resumability as first-class features**, not polish.
3. **Implement diagnostics before panel-review complexity** to reduce debugging cost.
4. **Add explicit YAML parsing for approval files** rather than relying only on string matching.
5. **Keep Step 11 isolated and heavily tested**, because it combines iteration control, severity routing, scoring, and final readiness decisions.
6. **Validate success criteria incrementally per milestone**, not only at the end.
