# Project Roadmap: CLI Portify Pipeline

## 1. Executive Summary

The CLI Portify system is an enterprise-grade pipeline that converts inference-based SuperClaude skill workflows into deterministic, programmatic CLI runners. It operates as a 12-step, 4-phase pipeline orchestrating Claude subprocess invocations with a gate enforcement system, convergence-loop panel review, and full resume/checkpoint support.

**Scope**: 65 requirements (47 functional, 18 non-functional) across 8 technical domains. Complexity score 0.92 (enterprise). The system produces release-quality specifications from skill definitions through automated analysis, specification synthesis, and expert panel review.

**Key architectural decisions**:
- Sequential-only pipeline (no parallel step groups) — simplifies executor but increases wall-clock time
- Domain models extend existing pipeline framework base types — tight coupling to framework but consistent API
- 12-gate enforcement with STANDARD/STRICT tiers — high correctness guarantees at the cost of implementation effort
- Convergence loop with 3-iteration cap — bounded resource consumption with graceful escalation

**Critical path**: Framework base types (D-008) must be stable before any domain model work begins. The `claude` CLI binary (D-001) is a hard runtime dependency with no fallback.

## 2. Phased Implementation Plan

### Phase A: Foundation & Prerequisites (Steps 0–1 equivalent)

**Goal**: Establish project skeleton, domain models, validation infrastructure, and component discovery.

#### Milestone A1: Project Scaffold & Domain Models
1. Create module directory at `src/superclaude/cli/cli_portify/`
2. Implement `models.py`: `PortifyConfig`, `PortifyStep`, `PortifyStepResult`, `PortifyOutcome`, `PortifyStatus`, `MonitorState`, `TurnLedger` — all extending pipeline base types (AC-003, NFR-016)
3. Implement `PortifyValidationError` with error codes: `NAME_COLLISION`, `OUTPUT_NOT_WRITABLE`, `AMBIGUOUS_PATH`, `INVALID_PATH`, `DERIVATION_FAILED` (FR-004 through FR-009)
4. Implement `__init__.py` with `"Generated by"` / `"Portified from"` marker (AC-013)

#### Milestone A2: Input Validation & Config Resolution
1. Implement `config.py`: workflow resolution via glob on `src/superclaude/skills/` (FR-001), CLI name derivation with `sc-` prefix and `-protocol` suffix stripping (FR-002, FR-003), name collision detection with portification marker check (FR-004), output directory validation (FR-005), working directory creation at `.dev/portify-workdir/<cli_name>/` (FR-006)
2. Emit `portify-config.yaml` with resolved paths
3. Implement timeout enforcement: 30s for input-validation (NFR-001)

#### Milestone A3: Component Discovery & Inventory
1. Implement `inventory.py`: scan `SKILL.md`, command `.md` files from both command directories, `refs/`, `rules/`, `templates/`, `scripts/`, `decisions.yaml` (FR-010)
2. Emit `component-inventory.yaml` with `{path, lines, purpose, type}` per component (FR-011)
3. Implement timeout enforcement: 60s for component-discovery (NFR-001)

**Exit criteria**: SC-001 and SC-002 pass. `portify-config.yaml` and `component-inventory.yaml` produced with valid content.

### Phase B: Gate System & Executor Core

**Goal**: Build the gate enforcement infrastructure and executor shell before any Claude-assisted steps.

#### Milestone B1: Gate Framework
1. Implement `gates.py`: all 12 gates (G-000 through G-011) with frontmatter field checks, minimum line counts, tier assignment (STANDARD/STRICT), and `GateMode.BLOCKING` enforcement (FR-045, FR-046)
2. Implement semantic check functions returning `tuple[bool, str]` (FR-047, AC-004)
3. Implement `EXIT_RECOMMENDATION` marker detection for gates G-002, G-003, G-005–G-008, G-010 (NFR-015)
4. Implement placeholder sentinel check (`has_zero_placeholders`) for G-010 (NFR-004)
5. Implement step-count consistency check for G-008 (NFR-013)
6. Implement return type pattern check for G-006 (NFR-014)

#### Milestone B2: Executor Shell
1. Implement `process.py`: Claude binary detection via `shutil.which("claude")` (FR-036, AC-008, NFR-017), subprocess launching with timeout wrapper, exit code classification (exit 124 → TIMEOUT, other non-zero → ERROR) (NFR-018, FR-041)
2. Implement `executor.py`: sequential step runner, `_determine_status()` logic based on exit code + `EXIT_RECOMMENDATION` + artifact presence (FR-041), retry mechanism with `retry_limit=1` for Claude-assisted steps (NFR-002), diagnostics collection on failure (FR-042)
3. Implement signal handlers: `SIGINT`/`SIGTERM` → complete current step result → set `INTERRUPTED` outcome (FR-039, NFR-003)
4. Implement `TurnLedger`: `can_launch()` check, budget exhaustion → `HALTED` outcome (FR-040)

#### Milestone B3: Observability
1. Implement `monitor.py`: `OutputMonitor` tracking output bytes, growth rate, stall seconds, events, line count, convergence iteration, findings count, placeholder count (NFR-009)
2. Implement `logging_.py`: dual logging to `execution-log.jsonl` (machine-readable) and `execution-log.md` (human-readable) (NFR-007)
3. Implement `tui.py`: `PortifyTUI` start/stop lifecycle with real-time progress via `rich` (NFR-008)

**Exit criteria**: Gate framework validates synthetic test artifacts correctly. Executor can run pure-programmatic steps end-to-end. Signal handling verified.

### Phase C: Claude-Assisted Steps & Review Gates

**Goal**: Implement all Claude-assisted analysis and specification steps with their prompts.

#### Milestone C1: Phase 1 Analysis Steps
1. Implement `prompts.py`: prompt builder for `protocol-mapping` step (FR-012) — extracts structured protocol map with step classification, dependency mapping, parallel groups, gate extraction
2. Implement prompt builder for `analysis-synthesis` step (FR-015) — synthesises protocol map + inventory into analysis report
3. Wire steps to produce `protocol-map.md` (FR-013) and `portify-analysis-report.md` (FR-016) with required YAML frontmatter
4. Enforce `EXIT_RECOMMENDATION` markers (FR-014, FR-017)
5. Enforce 600s timeout for both steps (NFR-001)

#### Milestone C2: User Review Gates
1. Implement `review.py`: `user-review-p1` as pure-programmatic pause writing `phase1-approval.yaml` with `status: pending` (FR-018)
2. Implement `user-review-p2` with spec validation checks (FR-025)
3. Implement `--resume` logic: check approval YAML for `status: approved` before proceeding (FR-019, FR-026)
4. Implement step-skip logic for `--resume <step-id>` (FR-038, NFR-010)

#### Milestone C3: Phase 2 Specification Steps
1. Implement prompt builder for `step-graph-design` → `step-graph-spec.md` (FR-020)
2. Implement prompt builder for `models-gates-design` → `models-gates-spec.md` (FR-021)
3. Implement prompt builder for `prompts-executor-design` → `prompts-executor-spec.md` (FR-022)
4. Implement `pipeline-spec-assembly`: programmatic pre-assembly (concatenation, dedup) + Claude synthesis → `portify-spec.md` (FR-023)
5. Enforce `EXIT_RECOMMENDATION` markers on all Phase 2 Claude steps (FR-024)
6. Enforce 600s timeout per step (NFR-001)

**Exit criteria**: SC-003 through SC-007 pass. Analysis and specification phases produce gate-passing artifacts. Resume works across review boundaries.

### Phase D: Release Spec Synthesis & Panel Review

**Goal**: Implement the highest-complexity steps: template-driven spec synthesis and convergence-loop panel review.

#### Milestone D1: Release Spec Synthesis (Step 10)
1. Load template from `src/superclaude/examples/release-spec-template.md` (AC-009, D-007)
2. Implement 4-substep synthesis: working copy creation (3a), 13-section population from Phase 1+2 outputs (3b), 3-persona brainstorm pass with `{gap_id, description, severity, affected_section, persona}` findings (3c), finding incorporation with unresolvable findings routed to Section 11 (3d) (FR-027)
3. Implement placeholder sentinel validation (FR-028)
4. Emit `portify-release-spec.md` with frontmatter `{title, status, quality_scores}` (FR-029)
5. Handle >50KB template via file argument passing (R-011)
6. Enforce 900s timeout (NFR-001)

#### Milestone D2: Panel Review Convergence Loop (Step 11)
1. Implement convergence state machine: `NOT_STARTED → REVIEWING → INCORPORATING → SCORING → CONVERGED|ESCALATED` (FR-030)
2. Implement per-iteration logic: 4-expert focus pass (Fowler, Nygard, Whittaker, Crispin) (4a), finding incorporation with severity routing (4b), full panel critique with quality scoring (4c), convergence scoring (4d) (FR-031)
3. Implement CONVERGED condition: zero unaddressed CRITICALs → `status: success` (FR-032)
4. Implement ESCALATED condition: 3 iterations exhausted → `status: partial` (FR-033)
5. Set `downstream_ready = true` when `overall >= 7.0` (FR-034, NFR-005)
6. Emit updated `portify-release-spec.md` and `panel-report.md` (FR-035)
7. Use internal convergence loop, NOT outer retry mechanism (AC-011)
8. Enforce 1200s timeout (NFR-001)

**Exit criteria**: SC-008 through SC-010 pass. Convergence loop reaches CONVERGED for well-formed inputs. ESCALATED path surfaces findings correctly.

## 3. Risk Assessment and Mitigation

### Critical Risks

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| **R-001**: Claude subprocess stall | Pipeline hangs indefinitely | Medium | Implement `stall_action: kill` with SIGTERM→SIGKILL escalation after `stall_timeout`. Monitor `growth_rate_bps` in `OutputMonitor`. Step timeouts as hard backstop. |
| **R-002**: Turn budget exhaustion before Steps 10–11 | No release spec produced | Medium | Front-load budget allocation for high-value steps. `TurnLedger.can_launch()` check with per-step cost estimates. `suggested_resume_budget` in contract. |
| **R-003**: Placeholder sentinels in final spec | Incomplete release spec | Low-Medium | Gate G-010 halts immediately. Diagnostics report remaining placeholders by name. Retry limit of 1 for Step 10. |
| **R-004**: Convergence ESCALATED | Partial spec, manual intervention needed | Medium | 3-iteration cap bounds cost. `panel-report.md` surfaces all unresolved CRITICALs. `downstream_ready = false` prevents silent propagation. |

### Moderate Risks

| Risk | Mitigation |
|------|------------|
| **R-005**: Ambiguous skill resolution | Early validation in Step 0 with candidate list in error message |
| **R-006**: Name collision | Portification marker detection; error before any work begins |
| **R-007**: Gate false-positives (line counts) | Retry limit of 1; prompts instruct comprehensive output; tunable thresholds |
| **R-008**: Approval file corruption | Simple string match is robust; consider adding YAML parse validation |
| **R-009**: Missing EXIT_RECOMMENDATION | `PASS_NO_SIGNAL` status is distinguishable; gates check for marker explicitly |

### Architectural Risk: Framework Base Type Stability (unlisted)

**Impact**: High. All domain models extend `PipelineConfig`, `Step`, `StepResult`, `GateCriteria`, `GateMode`, `SemanticCheck`. If these base types change, all domain models break.

**Mitigation**: Pin to current framework API. Add integration test that imports all base types. Document exact base type interface contract used.
