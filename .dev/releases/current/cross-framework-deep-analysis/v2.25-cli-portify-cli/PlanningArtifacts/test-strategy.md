# Comprehensive Test Strategy: CLI Portify Pipeline

## 1. Validation Milestones Mapped to Roadmap Phases

### M0 — Architecture Baseline
**Phase**: 0 (Architecture Confirmation)
**What to test**: Smoke-check that all pipeline framework base types are importable; dry-compile each open-question assumption into a concrete assertion before any implementation begins.

- Import assertions: `PipelineConfig`, `Step`, `StepResult`, `GateCriteria`, `GateMode`, `SemanticCheck` importable from current package
- Verify `SKILL.md` exists for at least one known workflow in `src/superclaude/skills/`
- Verify `src/superclaude/examples/release-spec-template.md` exists with all 13 sections and at least one `{{SC_PLACEHOLDER:*}}` sentinel
- Verify `shutil.which("claude")` returns non-None in the test environment (skip/xfail marker if not available)
- OQ resolution assertions: document confirmed answers for OQ-001, OQ-003, OQ-004, OQ-009, OQ-011 as constants in a `conftest.py` fixture

**Exit criterion**: All import assertions pass; release-spec-template structure validated; environment check fixtures stable.

## 2. Test Categories

### Unit Tests
**Scope**: Single module, all external dependencies mocked. No subprocess calls, no filesystem I/O beyond temp dirs.

**Modules covered**: `config.py`, `inventory.py`, `failures.py`, `models.py`, `executor.py`, `process.py`, `resume.py`, `contract.py`, `monitor.py`, `gates.py`, `review.py`, `tui.py`, `logging_.py`, `diagnostics.py`

**Characteristics**: Fast (<2s per test), deterministic, no network, no `claude` binary required. Use `pytest-mock` for `shutil.which`, subprocess calls, and signal delivery.

**Coverage target**: 90% line coverage on all non-Claude-invocation code paths.

### Integration Tests
**Scope**: Multiple modules working together; real filesystem using `tmp_path`; Claude subprocess mocked with a script that emits controlled output.

**Claude stub approach**: A shell script or Python subprocess that writes pre-defined output to stdout, exits with a configurable code, and optionally omits `EXIT_RECOMMENDATION` to test retry logic.

**Coverage**: Phase boundaries, gate enforcement end-to-end, resume logic, review gate pause/resume cycle.

**Characteristics**: Medium speed (2–30s per test), `pytest-timeout` applied at step level, real YAML written and parsed.

### End-to-End (E2E) Tests
**Scope**: Full pipeline execution against a real (minimal) skill fixture directory; real `claude` binary if available in CI, otherwise skipped via `pytest.mark.skipif(shutil.which("claude") is None, ...)`.

**Fixture skill**: Create `tests/fixtures/sc-test-portify-protocol/` with minimal `SKILL.md`, one command `.md`, and stub `refs/`, `rules/` content.

**Characteristics**: Slow (minutes), run only in CI nightly or on demand via `pytest -m e2e`. Always emit `return-contract.yaml`; always assert on its presence regardless of outcome.

### Acceptance Tests
**Scope**: The 14 success criteria as a formal test class. Run against M10 build only. These are the release gate — merge is blocked if any fail.

**Runner**: `uv run pytest tests/acceptance/test_success_criteria.py -v`

## 3. Test–Implementation Interleaving Strategy

**Ratio**: 1:2 — for every implementation phase, one test phase follows immediately before the next implementation phase begins. No phase is considered complete until its tests pass.

```
Phase 0 (Architecture)
  → Write M0 import/structure assertions
  → GATE: M0 assertions pass

Phase 1 (Config/Discovery)
  → Write M1 unit tests (FR-001–FR-011)
  → Implement config.py, inventory.py, failures.py
  → GATE: M1 tests pass

Phase 2 (Executor + Observability Baseline)
  → Write M2 unit tests (executor internals, base-type checks, observability baseline)
  → Implement models.py, executor.py, process.py, resume.py, contract.py, monitor.py (baseline), tui.py (skeleton), logging_.py (skeleton)
  → GATE: M2 tests pass

Phase 3 (Gate System)
  → Write M3 unit tests (all 12 gates, synthetic fixtures)
  → Implement gates.py
  → GATE: M3 tests pass

[Super-milestone A gate: SC-001, SC-002, SC-011, SC-014]

Phase 4 (Analysis Pipeline)
  → Write M4 integration tests (review gate 1, Claude stub)
  → Implement Phase 1 Claude steps, review gate, resume logic
  → GATE: M4 tests pass (SC-003, SC-004, SC-006, SC-007)

Phase 5 (Spec Pipeline)
  → Write M5 integration tests (review gate 2)
  → Implement Phase 2 Claude steps, spec assembly, review gate 2
  → GATE: M5 tests pass (SC-005)

Phase 6 (Release Spec Synthesis)
  → Write M6 unit/integration tests (placeholder detection, brainstorm, template loading)
  → Implement synthesis step
  → GATE: M6 tests pass (SC-008)

[Super-milestone B gate: SC-003–SC-008]

Phase 7 (Panel Review)
  → Write M7 state machine unit tests and convergence integration tests
  → Implement review.py
  → GATE: M7 tests pass (SC-009, SC-010)

Phase 8 (Observability Completion)
  → Write M8 diagnostics/TUI tests
  → Complete tui.py, monitor.py, diagnostics.py, logging_.py
  → GATE: M8 tests pass

Phase 9 (CLI Integration)
  → Write M9 CLI option tests
  → Implement commands.py, __init__.py, main.py wiring
  → GATE: M9 tests pass (SC-012, SC-013)

Phase 10 (Verification)
  → Write E2E and edge case tests
  → Run full SC regression matrix
  → GATE: All 14 SC pass

[Super-milestone C gate: SC-009–SC-013, full regression]
```

**Test-first rule**: Unit tests for each module's public API are written before implementation begins. Integration tests are written concurrently with the phase they cover, not after.

## 4. Risk-Based Test Prioritization

Tests are prioritized by the roadmap's risk inventory. High-severity risks get the deepest test coverage and run earliest.

| Priority | Risk | Test Focus |
|----------|------|-----------|
| P1 | R-001: Claude subprocess stall | `test_output_monitor_stall_detection`, timeout classification (exit 124 → TIMEOUT), stall-action kill invocation — these must pass before Phase 4 |
| P1 | R-002: Turn budget exhaustion | `TurnLedger` exhaustion unit tests, `suggested_resume_budget` formula, `HALTED` outcome contract emission — must pass at M2 |
| P1 | R-003: Placeholder leakage | G-010 `has_zero_placeholders` unit test with injected sentinels — must pass at M3 before Phase 6 runs |
| P1 | R-004: Escalation without downstream-ready | State machine ESCALATED path, `downstream_ready = false` assertion, `panel-report.md` content — must pass at M7 |
| P1 | R-013: Base type stability | Import assertions at M0 — blocks all subsequent work if they fail |
| P2 | R-007: Gate false positives | Line-count threshold tests with minimal-but-valid content; retry trigger test — M3 |
| P2 | R-008: Approval file corruption | YAML parse error surfaced (not silent string match) — M4 and M5 |
| P2 | R-009: EXIT_RECOMMENDATION absence | PASS_NO_SIGNAL classification test; gate marker check tests — M2 and M3 |
| P2 | R-005: Ambiguous skill resolution | `AMBIGUOUS_PATH` error with candidate list — M1 |
| P2 | R-006: Name collision | Marker-based collision detection — M1 |
| P3 | R-011: Template >50KB | File-argument path test — M6, single parametrize case |
| P3 | R-012: Log growth | Log scoped to workdir; no assertion needed beyond existence — M8 |

**CI fast-path**: P1 tests run on every commit. P2 tests run on every PR. P3 tests run in nightly suite.

### Gate Enforcement Mechanism

```bash
# Each gate is a named pytest marker group
uv run pytest -m "milestone_a0" --tb=short          # Gate A0
uv run pytest -m "milestone_a3" --tb=short          # Super-milestone A
uv run pytest -m "milestone_b6" --tb=short          # Super-milestone B
uv run pytest -m "milestone_c10" --tb=short         # Release gate
uv run pytest tests/acceptance/ -v                  # SC regression (release only)
```

CI pipeline runs gates in sequence; any gate failure blocks the pipeline and requires the implementing engineer to resolve before the next phase begins.
