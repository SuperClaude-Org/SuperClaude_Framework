## 1. Problem Statement

Two independent subprocess invocation failures prevent `superclaude` pipeline
commands from completing. Both failures only manifest in non-interactive
subprocess mode — the common theme being that subprocess invocations of `claude`
were under-configured relative to the interactive mode baseline.

**FIX-001**: `superclaude sprint run` fails consistently at Phase 2 because
`claude` subprocesses spawned in `--print --output-format stream-json` mode do
not have tool schemas available at session start. The model attempts to call
`TodoWrite` and `Bash` using guessed parameter shapes, which differ from the
actual schemas, causing `InputValidationError` on every invocation.

**FIX-ARG-TOO-LONG**: The roadmap pipeline crashes with
`OSError: [Errno 7] Argument list too long` at the `spec-fidelity` step when
combined input files exceed 128 KB. The pipeline's embed guard
(`_EMBED_SIZE_LIMIT = 200 KB`) is set above the Linux kernel's hard per-argument
limit (`MAX_ARG_STRLEN = 128 KB`), creating a 72 KB dead zone where content
passes the guard but crashes `subprocess.Popen`.

## 2. Solution Overview

Two independent, complementary fixes ship together in v2.24.5. Both address the
common theme of subprocess invocations being under-configured relative to
interactive mode.

**FIX-001** inserts `"--tools", "default"` into the command list built by
`ClaudeProcess.build_command()`. This flag tells the `claude` CLI to inject all
built-in tool schemas at session start, bypassing the message-history discovery
mechanism that fails in non-interactive subprocess mode. The change is a
two-line addition to one method; all subclasses inherit it automatically.

**FIX-ARG-TOO-LONG** replaces the miscalibrated `_EMBED_SIZE_LIMIT` constant
with a value derived from the OS constraint it is meant to respect, and changes
the embed guard to measure the actual string that becomes the `-p` argument
rather than just the embedded portion. A blocking Phase 0 empirical test
determines whether the existing `--file` fallback path works before any code is
changed.

**Interaction note**: `build_command()` in `pipeline/process.py` is the shared
delivery mechanism for both fixes. FIX-001's `--tools default` flag is present
in the subprocess command regardless of which prompt-delivery path
(inline `-p` or `--file` fallback) the ARG-TOO-LONG fix selects — the changes
are additive and non-conflicting. FIX-001 must be applied before the combined
test suite is run, as it shifts flag positions in the command list and any
index-based assertions must be updated first.

### 2.1 Key Design Decisions

#### FIX-001: Tool Schema Discovery

| Decision | Choice | Alternatives Considered | Rationale |
|----------|--------|------------------------|-----------|
| Where to add the flag | `pipeline/process.py:build_command()` (base class) | Each subclass independently; `extra_args` at call site | Single authoritative location; all consumers inherit; no call-site changes needed |
| Flag value | `--tools default` | `--tools all`, per-tool enumeration | `default` matches the built-in set actually needed; `all` may include experimental tools |
| Placement in cmd list | After `--no-session-persistence`, before `--max-turns` | End of list, before `-p` | Keeps session-control flags grouped; does not affect flag parsing order |

#### FIX-ARG-TOO-LONG: Embed Size Guard

| Decision | Choice | Alternatives Considered | Rationale |
|----------|--------|------------------------|-----------|\
| Limit derivation | `_MAX_ARG_STRLEN - _PROMPT_TEMPLATE_OVERHEAD` (~120 KB) | Hardcode 100 KB; query `getrlimit()` at runtime | Self-documenting derivation prevents future drift. `getrlimit` doesn't expose `MAX_ARG_STRLEN` (compile-time constant). |
| Guard measurement | Full composed string (`step.prompt + embedded`) | Embedded-only (current) | Closes template-overhead blind spot |
| Template overhead | 8 KB (2.3x largest template at 3.4 KB) | 4 KB (1.14x); dynamic measurement | Conservative margin justified by history of silent drift |
| `--file` fallback | Validate empirically before relying on it | Assume working; replace immediately | 80% likely broken per code review; 15-min test resolves ambiguity before committing to fix strategy |

### 2.2 Workflow / Data Flow

#### FIX-001: Tool Schema Discovery

```
BEFORE (broken):
  build_command() → [..., --no-session-persistence, --max-turns N, ...]
    → subprocess spawned → tool schemas NOT injected → model guesses schemas
    → InputValidationError on TodoWrite / Bash → Phase 2 exits code 1

AFTER (fixed):
  build_command() → [..., --no-session-persistence, --tools default, --max-turns N, ...]
    → subprocess spawned → tool schemas injected at init event
    → TodoWrite / Bash called with correct parameter shapes → Phase 2 succeeds
```

#### FIX-ARG-TOO-LONG: Embed Size Guard

```
BEFORE (broken):
  _embed_inputs(files) → len(embedded) <= 200KB? → YES → compose prompt (152 KB)
    → Popen("-p", 152KB_string) → execve() → KERNEL REJECTS (>128 KB) → OSError [Errno 7]

AFTER (fixed):
  _embed_inputs(files) → compose = prompt + embedded → len(composed) <= 120KB?
    → YES (inline path):
         executor.py sets extra_args = []
         ClaudeProcess.build_command() assembles:
           [..., --tools default, ..., -p <composed_prompt>] + []
         → Popen("-p", composed)  → execve() succeeds (< 128 KB)
    → NO (fallback path):
         executor.py sets extra_args = ["--file", path, ...]
         ClaudeProcess.build_command() assembles:
           [..., --tools default, ..., -p <base_prompt>] + ["--file", path, ...]
         → Popen("--file", paths...) → execve() succeeds (short args)

In both paths, --tools default is injected by build_command() unconditionally.
The two fixes are additive at the build_command() assembly level.
```

### FR-ATL.1: Derive `_EMBED_SIZE_LIMIT` from `MAX_ARG_STRLEN`

**Description**: Replace `_EMBED_SIZE_LIMIT = 200 * 1024` with a derivation from
`_MAX_ARG_STRLEN = 128 * 1024` minus `_PROMPT_TEMPLATE_OVERHEAD = 8 * 1024`.

**Acceptance Criteria**:
- [ ] `_MAX_ARG_STRLEN`, `_PROMPT_TEMPLATE_OVERHEAD`, and `_EMBED_SIZE_LIMIT` are defined as module-level constants in `executor.py`
- [ ] `_EMBED_SIZE_LIMIT` equals `_MAX_ARG_STRLEN - _PROMPT_TEMPLATE_OVERHEAD` (120 KB)
- [ ] No `import resource` line (dead code from brainstorm draft)
- [ ] Stale `# 100 KB` comment is removed
- [ ] A module-level `assert _PROMPT_TEMPLATE_OVERHEAD >= 4096` is present immediately after the three constant definitions, with an error message stating the kernel margin rationale and the measured template peak (~3.4 KB). The assertion must be triggered on any `import executor` call and therefore on every test run.
- [ ] Each of the three constants carries an inline comment: `_MAX_ARG_STRLEN` notes it is a Linux kernel compile-time constant; `_PROMPT_TEMPLATE_OVERHEAD` notes the 2.3x safety factor and the measured peak; `_EMBED_SIZE_LIMIT` notes its derivation and resulting byte value.

**Dependencies**: None

### FR-ATL.2: Guard full composed `-p` argument

**Description**: Change the embed guard to measure `step.prompt + "\n\n" + embedded`
(the actual value that becomes the `-p` argument), not just `embedded`.
The limit `_EMBED_SIZE_LIMIT = 120 KB` is derived from
`MAX_ARG_STRLEN - _PROMPT_TEMPLATE_OVERHEAD` (FR-ATL.1), so `<=` at 120 KB is
intentionally within the 128 KB kernel ceiling.

**Acceptance Criteria**:
- [ ] Guard evaluates `len(composed.encode("utf-8")) <= _EMBED_SIZE_LIMIT` where `composed = step.prompt + "\n\n" + embedded`
- [ ] Warning log reports "composed prompt" and actual byte count
- [ ] Fallback fires for inputs that fit individually but exceed limit when combined with prompt
- [ ] When `len(composed.encode('utf-8')) == _EMBED_SIZE_LIMIT` exactly (at-limit), inline embedding fires (not fallback) — confirming `<=` is the intended operator
- [ ] Implementation includes a code comment adjacent to the guard explaining: `<= is intentional; _EMBED_SIZE_LIMIT = 120 KB is safely below MAX_ARG_STRLEN = 128 KB`

**Dependencies**: FR-ATL.1

## 7. Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| `--tools default` enables a previously hidden tool, causing unintended model behaviour | Low | Medium | The set of default tools is stable and already used in interactive mode; subprocess mode was the anomaly |
| Subclass overrides `build_command()` without calling `super()` | Low | Low | Read subclass files before editing to confirm inheritance chain; none currently override. ARG-TOO-LONG's independent executors confirmed out of scope, limiting blast radius. |

| 8 KB overhead too conservative, causing unnecessary `--file` fallback for 112-120 KB inputs | Low | Low | Full-string measurement (FR-ATL.2) is the real safety gate; overhead is a soft pre-filter |
| Future prompt template exceeds 8 KB, narrowing the safe embed window | Low | Medium | `_PROMPT_TEMPLATE_OVERHEAD` is a named constant — easy to update if templates grow |
| Windows 32 KB command-line limit affects cross-platform users | Low | Medium | Noted for Phase 2; Linux-first fix is correct priority. `--tools default` adds two short tokens — negligible effect on Windows limit. |
| Executors not inheriting ClaudeProcess lack `--tools default` after Phase 1.5 activates | Low | Medium | Tracked as OQ-4; assess scope after Phase 0 result is known |
