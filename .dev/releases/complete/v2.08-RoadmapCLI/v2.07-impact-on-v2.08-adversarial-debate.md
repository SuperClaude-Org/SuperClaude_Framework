## Round 1: Test Baseline Integrity

### Advocate Position

The v2.08 spec makes a critical assumption in deliverable D2.4 and milestone D5.3: "All 14 existing sprint test files must pass without modification." This exact phrase appears in both the spec (NFR-002) and roadmap (D2.4, D5.3). It is repeated as AC-06 and in the "Sprint test stability guarantee" section at line 1094 of the merged spec.

This assumption is now demonstrably false. v2.07 modified 9 out of 14 test files with 394 insertions and 48 deletions. These are not trivial changes:

1. **test_models.py** (+34/-6): Tests now expect `PhaseStatus.SKIPPED` to exist and be terminal. The old tests expected it was removed. Any migration that doesn't account for SKIPPED will break.

2. **test_config.py** (+47/-6): Tests the new `validate_phases()` return type (`dict` with "errors"/"warnings" keys instead of a flat `list`). Any extraction that refactors config.py must preserve this new API.

3. **test_executor.py** (+232/-1): Massively expanded with watchdog and stall detection tests. These tests import from `debug_logger` and `diagnostics` -- two entirely new modules that the v2.08 spec does not account for.

4. **test_process.py** (+53/-1): Tests now assert `--verbose` and `--output-format stream-json` in the command builder, and test the new `env.pop()` behavior.

The v2.08 spec's success criterion SC-006 ("all existing sprint tests pass after migration") now has a moving target. The "existing" tests as of v2.08 implementation will be the v2.07-modified tests, not the tests the spec was written against. The spec's language about "no test file modifications" is retroactively invalidated.

### Critic Response

The Advocate conflates the intent of the spec with its literal phrasing. The v2.08 spec was written on 2026-03-04, before v2.07 was merged. The intent of NFR-001/002 and AC-06 is clear: **after the pipeline/ extraction, sprint must continue to work exactly as it did before the extraction began**. This is a regression guarantee, not a time-frozen snapshot guarantee.

When v2.08 implementation begins, the sprint codebase will be whatever state v2.07 left it in. The "existing" tests at implementation time ARE the v2.07-modified tests. The spec's regression guarantee means: "extract pipeline/ and migrate sprint, and all tests that pass before you start must still pass after you finish."

The phrase "no test file modifications" means the v2.08 implementer must not modify sprint tests to make migration work -- it does not mean tests can never be modified by prior releases. This is a one-directional constraint: v2.08 must not break what v2.07 established.

Regarding test_executor.py importing `debug_logger` and `diagnostics`: these are sprint-specific modules. The v2.08 extraction plan explicitly states "Sprint-specific orchestration (TUI, tmux, stall detection, phase numbering) stays in SprintConfig." The new diagnostic modules stay in sprint/. The executor stays in sprint/ (only `execute_pipeline()` is a new generic function). The test imports resolve correctly because nothing moves these files.

### Round 1 Synthesis

**Convergence**: 70%. Both sides agree the literal text is outdated but disagree on materiality.

- **Agreed**: The spec's phrasing about "14 existing test files" and "no modification" is technically inaccurate as written and should be updated for clarity.
- **Disagreed**: Whether this requires substantive spec changes or merely editorial updates.
- **Key insight**: The regression guarantee's intent is preserved -- the question is whether the extraction plan accounts for the changed code it will be extracting from.

## Round 2: Extraction Complexity -- ClaudeProcess and Output Format

### Advocate Position

The v2.07 changes to `process.py` are not cosmetic. They fundamentally alter ClaudeProcess behavior in ways that affect the pipeline/ extraction:

**1. Output format change: `text` -> `stream-json`**

ClaudeProcess now passes `--output-format stream-json` and `--verbose` to the claude subprocess. The v2.08 spec (section 13.1) plans to move ClaudeProcess to `pipeline/process.py` as a shared component. But the spec's roadmap command needs ClaudeProcess to produce gate-checkable output files. The spec says (line 164): "`gate_passed()` in `pipeline/gates.py`" validates output files.

If ClaudeProcess writes stream-json NDJSON to its stdout file, the gate validation must either:
- Parse NDJSON to extract the actual text content before checking YAML frontmatter and line counts, OR
- Configure ClaudeProcess to use a different output format per consumer

Neither scenario is addressed in the spec. The spec assumes output files are plain markdown with YAML frontmatter (see gate criteria in section 3.2). But the ClaudeProcess the spec plans to extract now produces NDJSON, not plain text.

**2. Environment variable handling**

The change from `env["CLAUDECODE"] = ""` to `env.pop("CLAUDECODE", None)` and the addition of `env.pop("CLAUDE_CODE_ENTRYPOINT", None)` means the shared ClaudeProcess has hardcoded assumptions about the execution environment. This is fine for both sprint and roadmap, but the spec doesn't document this behavior.

**3. Monitor coupling**

The monitor.py rewrite to NDJSON parsing is tightly coupled to the `stream-json` output format. When ClaudeProcess moves to pipeline/, the monitor stays in sprint/. But if roadmap needs its own monitoring (even basic progress tracking), it cannot reuse sprint's monitor because it's NDJSON-specific. The spec doesn't address how roadmap will observe subprocess progress.

The cumulative effect: the ClaudeProcess that the spec planned to extract is materially different from the one the spec was written against. The spec describes a text-output ClaudeProcess; the actual ClaudeProcess produces stream-json.

### Critic Response

Point by point:

**1. Output format**: The Advocate is correct that this is a material change, but overstates its impact. The `--output-format` flag is a ClaudeProcess parameter. When extracting to pipeline/, the output format becomes configurable per-consumer. Sprint passes `stream-json`; roadmap can pass `text`. The ClaudeProcess class already parameterizes everything else (max_turns, model, permission_flag). Adding output_format as a parameter is a trivial change within the extraction -- it does not invalidate the extraction plan.

Furthermore, examining the actual spec section 13.5, the spec describes a `StepRunner` protocol where each consumer provides its own subprocess lifecycle. The spec explicitly chose composition over inheritance BECAUSE "sprint's tightly-coupled orchestration (TUI, tmux, stall detection) cannot be meaningfully abstracted into hooks." The roadmap command defines its own `roadmap_run_step` function (D3.5) that builds subprocess argv. This function can specify any output format.

**2. Environment variable handling**: This is a non-issue. Both sprint and roadmap need to prevent nested session detection. The `env.pop()` approach is correct for both consumers. Whether this is documented or not is an editorial concern, not a design concern.

**3. Monitor coupling**: The spec explicitly puts monitoring out of scope for v1 roadmap. Spec section "Out of Scope (v1)" says: "TUI / rich progress display (stdout logging only)." The roadmap command uses simple stdout logging per progress display deliverable D4.3. It does not need sprint's monitor. This is a non-impact.

The actual extraction path is: move ClaudeProcess to pipeline/, make output_format a constructor parameter (defaulting to `stream-json` for backward compatibility), and let each consumer specify its format. The spec's composition-via-StepRunner architecture already accommodates this.

### Round 2 Synthesis

**Convergence**: 60%. Substantive disagreement on whether the output format change requires spec updates.

- **Agreed**: ClaudeProcess now produces stream-json, which differs from the spec's implicit assumption of text output. The extraction must account for this.
- **Agreed**: The environment variable change is immaterial.
- **Agreed**: Monitor coupling is not a v1 roadmap concern.
- **Disagreed**: Whether the output format change requires spec changes or is handled naturally by the extraction's parameterization.
- **Key insight**: The spec's `StepRunner` composition pattern already provides the mechanism for per-consumer output format. But the spec should explicitly document that `ClaudeProcess` output format is parameterized, because the spec's gate validation section assumes plain markdown files.

## Round 3: SprintConfig Shape, New Modules, and Extraction Boundary

### Advocate Position

The v2.08 spec's section 13.1 (PipelineConfig Field Allocation) was written with careful analysis of which fields belong in PipelineConfig vs. SprintConfig. This analysis IS ALREADY CORRECT for the v2.07 changes -- the spec lists `stall_timeout`, `stall_action`, `phase_timeout` as SprintConfig-only. This appears to account for v2.07's additions.

However, there are three areas where the spec fails to account for v2.07's actual implementation:

**1. New modules: `debug_logger.py` and `diagnostics.py`**

v2.07 added two entirely new modules to `sprint/`. The executor.py now imports from both:
```python
from .debug_logger import debug_log, setup_debug_logger
from .diagnostics import DiagnosticCollector, FailureClassifier, ReportGenerator
```

The v2.08 spec says `debug: bool` goes into PipelineConfig (line 567: "Both pipelines support debug logging"). But the debug logging IMPLEMENTATION lives in `sprint/debug_logger.py`. If debug is a PipelineConfig field, there should be a `pipeline/debug_logger.py` or equivalent -- but the spec does not mention this module at all.

**2. validate_phases() API change**

The return type changed from `list[str]` to `dict[str, list[str]]`. This is a sprint-internal function, but it signals that sprint's API surface is evolving. The extraction must work against the current API, not the API the spec was written against.

**3. SKIPPED status**

v2.07 re-added `PhaseStatus.SKIPPED` which had been intentionally removed (the old code had a comment explaining why). The v2.08 spec's `StepStatus` enum (in PipelineConfig) includes `SKIPPED`. If `PhaseStatus` is supposed to extend `StepStatus`, there could be a naming/semantics collision where SKIPPED means different things in the pipeline vs. sprint contexts.

**Net assessment**: The spec needs targeted updates in 3 areas: (a) document the debug logging module extraction or sprint-only retention, (b) update test baseline language, (c) note that ClaudeProcess output format is now stream-json and must be parameterized.

### Critic Response

Let me address each point and then give my final assessment:

**1. debug_logger.py**: The Advocate is right that this creates a minor gap. But the resolution is straightforward: `debug_logger.py` stays in sprint/. The `debug: bool` field in PipelineConfig controls WHETHER to log; the logging implementation is consumer-specific. Sprint uses `debug_logger.py`; roadmap writes to its own debug file. The spec does not need to describe sprint's internal debug logging module -- it just needs to note that debug logging implementation is per-consumer, not shared.

**2. validate_phases() API change**: This is entirely sprint-internal. `validate_phases()` is called only from sprint code. The pipeline/ extraction does not touch this function. Non-impact.

**3. SKIPPED status**: The spec already includes SKIPPED in `StepStatus` (line 134: `SKIPPED = "skipped"`). The v2.07 re-addition of `PhaseStatus.SKIPPED` actually ALIGNS sprint with the spec's planned pipeline `StepStatus`. When `PhaseStatus` extends `StepStatus`, SKIPPED will be inherited naturally. This is not a collision -- it is convergence. v2.07 accidentally made the extraction easier.

**My final position**: The v2.07 changes are materially compatible with the v2.08 extraction plan. The spec's architecture (composition via StepRunner, PipelineConfig/SprintConfig inheritance, one-directional dependency) already accommodates v2.07's additions. Three editorial updates are warranted:

1. Update test baseline language from "14 existing test files, no modifications" to "all sprint tests passing at time of extraction"
2. Note that ClaudeProcess output format changed to stream-json and should be parameterized in pipeline/
3. Clarify that debug logging implementation is per-consumer (sprint/debug_logger.py stays in sprint/)

None of these require structural changes to the milestone plan, dependency graph, or deliverable definitions.

### Round 3 Synthesis

**Convergence**: 85%. Strong agreement on the nature and scope of needed changes.

- **Agreed**: Three specific editorial/documentation updates are needed in the spec.
- **Agreed**: The milestone structure (M1-M5) and dependency graph remain valid.
- **Agreed**: SKIPPED status re-addition is convergent, not conflicting.
- **Agreed**: validate_phases() change is sprint-internal and immaterial.
- **Minor disagreement**: Whether "parameterize output format" is editorial (Critic) or a deliverable change (Advocate). The Critic's argument is stronger: the StepRunner composition pattern already handles this, so it is an implementation detail within existing deliverables, not a new deliverable.

### 2. PERCEIVED Impacts That Are Actually Manageable Without Spec Changes

| Perceived Impact | Why It Is Manageable |
|-----------------|---------------------|
| **"Sprint code has been heavily modified"** | The 431 insertions are predominantly additive (new features, expanded monitoring). The core patterns the spec plans to extract (subprocess lifecycle, config dataclass, process management) are structurally intact. |
| **"New modules (debug_logger, diagnostics) add extraction complexity"** | These modules stay in sprint/. The extraction does not touch them. Their imports from executor.py are sprint-side, not pipeline-side. |

| **"Regex rewrite in config.py"** | Sprint-internal phase discovery logic. Pipeline/ does not extract or depend on phase discovery. |
| **"Monitor rewrite to NDJSON"** | Sprint-specific. Monitor stays in sprint/. Roadmap v1 uses stdout logging, not a monitor. |
| **"Watchdog/stall detection added to executor"** | Sprint-specific. The v2.08 spec already planned to create a separate `execute_pipeline()` as a generic executor, distinct from `execute_sprint()`. The watchdog code stays in `execute_sprint()`. |

### 3. Roadmap Disposition: Incremental Updates

The v2.08 roadmap does **NOT** need refactoring. The milestone structure (M1-M5), dependency graph, deliverable definitions, and risk register remain valid. The changes required are:

- **3 editorial updates** to existing spec text (test baseline language, output format note, debug logging clarification)
- **0 structural changes** to milestones, dependencies, or deliverables
- **0 new deliverables** required
- **0 new risks** to add to the risk register

The v2.08 spec's architectural decisions (composition via StepRunner, PipelineConfig inheritance, one-directional dependency) were well-designed and naturally accommodate v2.07's changes.
