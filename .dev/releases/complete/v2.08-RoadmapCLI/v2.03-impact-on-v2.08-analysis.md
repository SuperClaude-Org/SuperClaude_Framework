# v2.03 Impact Analysis on v2.08 (RoadmapCLI)

**Date**: 2026-03-05
**Analyst**: System Architect
**v2.03**: Sprint CLI Diagnostic Testing Framework (spec-sprint-diagnostic-framework.md)
**v2.08**: `superclaude roadmap` CLI Command (merged-spec.md)

---

## Executive Summary

v2.03 specified a comprehensive diagnostic framework for sprint execution: debug logging, stall detection/watchdog, failure classification, and graduated test infrastructure. **The implementation is substantially complete** -- all core production modules are delivered and integrated. The test infrastructure is partially delivered. The implementation has **3 REAL IMPACTS** and **2 PERCEIVED IMPACTS** on v2.08. The primary concern is that diagnostic modules (`debug_logger.py`, `diagnostics.py`) are sprint-specific but contain patterns that v2.08's `pipeline/` extraction needs to account for.

## B. Impact on v2.08

### REAL IMPACT 1: `debug` Field Must Move to PipelineConfig

**Finding**: v2.08 merged-spec section 13.1 already specifies `debug: bool` as a `PipelineConfig` field, stating: "Both pipelines support debug logging. PipelineConfig provides the flag; logging infrastructure is consumer-specific (sprint owns `debug_logger.py`, roadmap provides its own)."

However, the current implementation has `debug` as a field on `SprintConfig` (line 96 of `models.py`), along with `debug_log_path` as a property on `SprintConfig` (line 102). During M2 (Sprint Migration to Pipeline), this field needs to move up to `PipelineConfig`.

**Impact**: The v2.08 spec already accounts for this correctly. The `debug_log_path` property, however, is sprint-specific (uses `self.results_dir`) and should NOT move to PipelineConfig. Roadmap will need its own path logic.

- **Classification**: REAL IMPACT
- **Severity**: Low
- **v2.08 Spec Section**: Section 13.1 (PipelineConfig Field Allocation)
- **Current text**: The table correctly shows `debug: bool` in PipelineConfig.
- **Recommended change**: Add a row clarifying `debug_log_path` remains in `SprintConfig` as a property, and roadmap creates its own path property. Also note that `setup_debug_logger()` and `debug_log()` stay in `sprint/debug_logger.py` -- roadmap may reuse or create its own logger setup.

### REAL IMPACT 2: `diagnostics.py` Complicates the Extraction Boundary

**Finding**: The v2.08 spec's module structure (section 3.1) lists sprint modules as:
```
sprint/
  commands.py, config.py, executor.py, models.py, process.py,
  monitor.py, tui.py, tmux.py, logging_.py, notify.py
```

This does **not** list `sprint/debug_logger.py` or `sprint/diagnostics.py`, which are now part of the sprint codebase. The executor imports both:
```python
from .debug_logger import debug_log, setup_debug_logger
from .diagnostics import DiagnosticCollector, FailureClassifier, ReportGenerator
```

During M2 (Sprint Migration), when `sprint/executor.py` is refactored to wrap `execute_pipeline()`, the question arises: should diagnostic collection be handled at the pipeline level or remain sprint-specific?

**Analysis**: The diagnostic framework is sprint-specific for v1:
- `DiagnosticCollector` reads sprint-specific artifacts (debug.log, output/error files by phase number)
- `FailureClassifier` uses sprint-specific heuristics (watchdog-triggered, stall-based)
- `ReportGenerator` produces sprint-formatted markdown
- Roadmap has its own failure policy (retry-then-halt with gate failure messages per spec section 6.2)

The diagnostic modules should stay in `sprint/` and not move to `pipeline/`. The pipeline executor provides its own failure reporting via `StepResult.gate_failure_reason`.

- **Classification**: REAL IMPACT
- **Severity**: Medium
- **v2.08 Spec Section**: Section 3.1 (Module Structure) and Section 12 (Sprint Migration Strategy)
- **Current text**: Module listing omits `debug_logger.py` and `diagnostics.py` from sprint/.
- **Recommended change**: Update section 3.1 module listing to include:
  ```
  sprint/
    ...
    debug_logger.py          <- v2.03: crash-safe debug logging (sprint-specific)
    diagnostics.py           <- v2.03: failure classification and reporting (sprint-specific)
  ```
  Update section 12 (Sprint Migration Strategy) step 5 to note: "sprint/executor.py continues to import and use debug_logger and diagnostics modules; these remain sprint-specific and do not move to pipeline/."

### REAL IMPACT 3: Monitor NDJSON Architecture Affects ClaudeProcess Extraction

**Finding**: v2.08 spec D1.4 says `ClaudeProcess` is extracted from `sprint/process.py` to `pipeline/process.py` with an `output_format` parameter (default `stream-json` for sprint, `text` for roadmap).

The current `ClaudeProcess` (process.py line 94) hardcodes `--output-format stream-json` in `build_command()`. The monitor (monitor.py) is built entirely around NDJSON parsing -- it splits on newlines, `json.loads()` each line, and extracts signals from structured events.

This is significant because:
1. The `output_format` parameterization in pipeline/process.py is correct and necessary.
2. The roadmap pipeline does NOT need the NDJSON monitor -- it uses gate validation on output files instead.
3. The sprint's `OutputMonitor` is tightly coupled to `stream-json` output format.

**The v2.08 spec already handles this correctly** -- monitor.py stays in sprint/ and is listed as "unchanged". But the spec should explicitly note that `ClaudeProcess.build_command()` must be parameterized to accept `output_format` rather than hardcoding it.

- **Classification**: REAL IMPACT
- **Severity**: Low
- **v2.08 Spec Section**: D1.4 (pipeline/process.py)
- **Current text**: "output_format parameter added (default: stream-json for sprint backward-compat; roadmap specifies text for gate-compatible output)"
- **Recommended change**: Note that the current `build_command()` (process.py line 93-94) hardcodes `"--output-format", "stream-json"` and this must be changed to use the `output_format` parameter. Also note that sprint's `OutputMonitor` depends on `stream-json` format -- if `output_format` is changed for sprint, the monitor breaks. This is a migration safety concern.

### PERCEIVED IMPACT 1: Debug Logger Pattern Could Be Shared

**Finding**: One might think `debug_logger.py` should move to `pipeline/` since roadmap also has a `--debug` flag (spec section 5). However, the spec correctly states "logging infrastructure is consumer-specific."

**Why it's only perceived**: The debug logger's implementation is sprint-specific in non-obvious ways:
- Logger name `superclaude.sprint.debug` and child loggers (`*.executor`, `*.process`, `*.monitor`, `*.tui`)
- Debug log path derived from `SprintConfig.results_dir`
- Event vocabulary (PHASE_BEGIN, poll_tick, etc.) is sprint-specific

If roadmap wants debug logging, it should create its own logger with its own event vocabulary. The `_FlushHandler` and `_DebugFormatter` classes could theoretically be shared, but YAGNI -- roadmap v1 is simpler and may not need crash-safe logging.

- **Classification**: PERCEIVED IMPACT (no spec change needed)

### PERCEIVED IMPACT 2: Watchdog/Stall Detection Might Affect Pipeline Extraction

**Finding**: The watchdog mechanism (stall_timeout, stall_action, single-fire guard) is deeply embedded in the sprint executor's poll loop (executor.py lines 126-162). One might worry this complicates extracting `execute_pipeline()`.

**Why it's only perceived**: The v2.08 spec explicitly keeps `stall_timeout`, `stall_action`, and `phase_timeout` in `SprintConfig` (section 13.1). The generic `execute_pipeline()` does not implement stall detection -- that remains in sprint's wrapper around the pipeline executor. The roadmap command uses per-step timeouts and gate validation instead of stall detection.

- **Classification**: PERCEIVED IMPACT (spec already handles this)

### NO IMPACT Items

| Item | Why No Impact |
|---|---|
| Diagnostic test directory (`tests/sprint/diagnostic/`) | Sprint tests are sprint-specific. v2.08 creates `tests/pipeline/` and `tests/roadmap/` separately. |
| FailureCategory enum simplification (5 vs 6 modes) | Sprint-internal; roadmap uses `StepStatus` (PASS/FAIL/TIMEOUT/SKIPPED/CANCELLED) not FailureCategory. |
| Report format (markdown vs JSON) | Sprint-internal diagnostic reports are not consumed by pipeline or roadmap. |
| TUI debug instrumentation | TUI stays in sprint/; roadmap uses stdout-only progress display. |
| Tmux flag forwarding | Tmux stays in sprint/; roadmap has no tmux support. |
| SprintConfig backward-compatible defaults | The defaults work correctly; migration just needs to move shared fields up to PipelineConfig. |

## C. Risk Assessment

### Risks Introduced by v2.03

| Risk | Severity | Description | Mitigation |
|---|---|---|---|
| **Extraction boundary complexity** | Medium | v2.03 added 2 new modules (`debug_logger.py`, `diagnostics.py`) and integrated them into `executor.py` with 6 import statements. This makes the sprint executor more complex to refactor during M2 (Sprint Migration). | During migration, keep these imports in sprint's executor wrapper. The generic `execute_pipeline()` should not import or use sprint diagnostics. |
| **debug_log() coupling** | Low | `debug_log()` is called from 5 modules (executor, process, monitor, tui, diagnostics). Moving `process.py` to `pipeline/process.py` means the pipeline module would need to import from sprint's debug_logger, violating NFR-07 (no sprint imports in pipeline). | Solution: `pipeline/process.py` should NOT include debug_log calls. Sprint re-exports `ClaudeProcess` from pipeline and can wrap or monkey-patch if debug logging is needed, OR accept that process-level debug logging is lost in the pipeline version (sprint-specific instrumentation stays in sprint's wrapper). |
| **Monitor stall_seconds dependency** | Low | The watchdog in executor.py reads `ms.stall_seconds` and `ms.events_received` from `MonitorState`. This couples the executor to monitor's NDJSON-based liveness tracking. If the monitor changes, the watchdog breaks. | Not a v2.08 risk -- monitor stays in sprint/. But it means sprint's executor can't fully delegate to `execute_pipeline()` without reimplementing the poll loop. |

### Risks Mitigated by v2.03

| Risk | Description |
|---|---|
| **Silent stall visibility** | v2.03 solved the core problem of sprints stalling with no output. If v2.08 migration introduces a regression that causes stalls, the diagnostic framework will catch it -- `--debug` and `--stall-timeout` provide observability. |
| **Failure root cause analysis** | During sprint migration testing (M2), if sprint tests fail, the diagnostic reports will help identify whether the failure is a migration issue vs a pre-existing problem. |
| **Backward compatibility verification** | The v2.03 diagnostic tests in `tests/sprint/diagnostic/` provide additional coverage during migration -- any breakage in the diagnostic pathway signals a regression. |

## D. Change Recommendations

### Recommendation 1: Update Module Listing (Medium Severity)

- **Section**: 3.1 (Module Structure)
- **Current**: Sprint module listing shows 10 files (commands, config, executor, models, process, monitor, tui, tmux, logging_, notify)
- **Change**: Add the two v2.03 modules:
  ```
  sprint/
    ...
    debug_logger.py          <- crash-safe debug logging (stays in sprint/)
    diagnostics.py           <- failure classification and reporting (stays in sprint/)
  ```
- **Rationale**: Accurate module listing prevents confusion during M2 implementation. Developers need to know these files exist and should NOT be moved to pipeline/.

### Recommendation 2: Clarify ClaudeProcess Debug Logging Migration (Low Severity)

- **Section**: 12 (Implementation Notes, Sprint Migration Strategy) step 2
- **Current**: "Move ClaudeProcess from sprint/process.py to pipeline/process.py; re-export from sprint/process.py"
- **Change**: Add note: "Remove debug_log() calls from pipeline/process.py. The pipeline version of ClaudeProcess must not import from sprint/debug_logger (NFR-07 violation). Sprint can add debug logging by wrapping ClaudeProcess methods or via a subclass if needed. Alternatively, accept that process-level debug logging is sprint-only and emitted from sprint/executor.py rather than from within ClaudeProcess itself."
- **Rationale**: The current `process.py` has 5 `debug_log()` calls. Moving the file to pipeline/ without addressing these creates an import cycle.

### Recommendation 3: Add output_format to ClaudeProcess.build_command Migration Note (Low Severity)

- **Section**: D1.4 description or Section 12 step 2
- **Current**: Mentions output_format parameter but doesn't note the existing hardcoded value.
- **Change**: Add: "Current build_command() hardcodes `--output-format stream-json` (process.py line 93-94). Parameterize this using a new `output_format: str = 'stream-json'` field on PipelineConfig or as a constructor parameter on ClaudeProcess. Sprint's OutputMonitor requires stream-json; roadmap gates require text output. Changing the default would break sprint's monitor."
- **Rationale**: Prevents a subtle regression where sprint's monitor stops working because `output_format` default was changed during extraction.

### Recommendation 4: Document Diagnostic Module Non-Extraction (Low Severity)

- **Section**: 12 (Sprint Migration Strategy) -- add new step between current steps 4 and 5
- **Current**: No mention of diagnostic modules during migration.
- **Change**: Add step: "Note: sprint/debug_logger.py and sprint/diagnostics.py (added in v2.03) remain in sprint/. They are not extracted to pipeline/. The sprint executor continues to import and use them. pipeline/executor.py provides its own failure reporting via StepResult.gate_failure_reason."
- **Rationale**: Explicit non-extraction decisions are as important as extraction decisions for implementation clarity.
