# v2.05 Impact Analysis on v2.08 (RoadmapCLI)

**Date**: 2026-03-05
**Analyst**: System Architect
**Branch**: v2.05-sprint-cli-specifications

---

## Executive Summary

v2.05 specified the `superclaude sprint` CLI from scratch. The sprint code is now fully implemented and has been further modified by v2.07 (tasklist work). v2.08 plans to extract a shared `pipeline/` module from the sprint code and build `superclaude roadmap` on top of it. This analysis identifies where the actual implementation state of sprint diverges from what v2.08 assumes, and what risks that creates.

**Key finding**: The v2.05 spec is substantially implemented. v2.07 introduced post-spec changes (debug logging, diagnostics, stall watchdog, stream-json output format) that v2.08 acknowledges in some places but not all. The primary risk is that v2.08's pipeline extraction plan references v2.05-era code structure while the actual code has evolved.

## B. Impact Classification on v2.08

### B.1 REAL IMPACT: Requires v2.08 Spec Change

#### IMPACT-1: ClaudeProcess output_format parameterization already partially done

**Finding**: v2.08 spec (merged-spec.md, D1.4) says `ClaudeProcess` needs an `output_format` parameter added, with sprint defaulting to `stream-json` and roadmap using `text`. In the actual code, `process.py` L94 hardcodes `stream-json`. The parameter does not yet exist as a configurable field -- it is a literal string in `build_command()`.

**Classification**: REAL IMPACT

**Why it matters**: v2.08's D1.4 deliverable assumes extracting ClaudeProcess to pipeline/ and adding an `output_format` parameter. The actual code structure is correct for this extraction, but the spec should note that the current hardcoded value must be replaced with a parameter, not that an existing parameter needs its default changed.

**Severity**: Low

**Recommendation**: In v2.08 roadmap D1.4, change acceptance criteria from "output_format parameter added (default: stream-json for sprint backward-compat)" to "output_format parameter replaces hardcoded 'stream-json' in build_command() (default: stream-json for sprint backward-compat; roadmap specifies text)".

---

#### IMPACT-2: SprintConfig has diagnostic fields not accounted for in PipelineConfig allocation

**Finding**: v2.08's PipelineConfig field allocation table (merged-spec.md Section 13.1) correctly lists `debug` in PipelineConfig and `stall_timeout`, `stall_action`, `phase_timeout` in SprintConfig. However, the actual SprintConfig also has `debug_log_path` and `results_dir` as computed properties that depend on `release_dir`. When SprintConfig inherits from PipelineConfig and `release_dir` becomes a property alias for `work_dir`, these computed properties will need to chain correctly through the alias.

**Classification**: REAL IMPACT

**Why it matters**: `SprintConfig.debug_log_path` returns `self.results_dir / "debug.log"`, and `results_dir` returns `self.release_dir / "results"`. If `release_dir` becomes a property aliasing `work_dir`, the chain `self.release_dir -> self.work_dir -> Path` must be set up before `debug_log_path` or `results_dir` are accessed. With Python dataclasses + inheritance + property aliasing, initialization order matters.

**Severity**: Medium

**Recommendation**: Add to v2.08 roadmap M2 deliverables: "Verify SprintConfig computed properties (debug_log_path, results_dir, execution_log_jsonl, execution_log_md, output_file, error_file, result_file) resolve correctly through the release_dir->work_dir property alias chain. Add regression tests for all 7 computed paths."

---

#### IMPACT-3: Sprint executor imports 2 files not in v2.05 spec

**Finding**: `executor.py` imports from `debug_logger` (L11) and `diagnostics` (L12). These are v2.07 additions. v2.08's Section 13.5 shows sprint/executor.py being refactored to call `execute_pipeline()` internally with a `sprint_run_step` callable. The diagnostic collection logic (executor.py L241-258) and debug logging (L53-56, L90, etc.) are tightly integrated into the executor's poll loop and post-failure logic.

**Classification**: REAL IMPACT

**Why it matters**: When refactoring `execute_sprint()` to wrap `execute_pipeline()`, the diagnostic collection and debug logging calls must be preserved inside the `sprint_run_step` callable or in the `on_step_complete` callback. The v2.08 spec's example code (Section 13.5) shows a simplified sprint_run_step that does not account for these integrations.

**Severity**: Medium

**Recommendation**: Update v2.08 Section 13.5's sprint_run_step example to note: "sprint_run_step must also integrate: (a) debug_logger calls per poll tick and phase boundaries, (b) DiagnosticCollector/FailureClassifier/ReportGenerator on failure status, (c) TUI error resilience wrapping. The ~100 lines referenced in the spec are approximately ~180 lines in practice due to v2.07 additions."

---

#### IMPACT-4: Monitor state model diverged from v2.05 byte-based design

**Finding**: v2.08 merged-spec.md does not explicitly discuss the monitor, stating sprint modules `monitor.py` and `tui.py` are "unchanged" (Section 3.1). But the monitor already changed from v2.05's byte-growth stall detection to event-based stall detection (tracking `last_event_time` and `events_received`). This is relevant because v2.08 needs `ClaudeProcess` output_format to be parameterizable -- sprint uses `stream-json` (generating NDJSON events the monitor parses), while roadmap will use `text` (plain text output).

**Classification**: REAL IMPACT

**Why it matters**: If roadmap ever adds monitoring (post-v1), the monitor's event-based design will not work with `text` output format. This is not a v1 blocker but it constrains future extensibility. More immediately, when extracting `ClaudeProcess` to pipeline/, the output_format parameterization directly affects which monitor patterns work. The v2.08 spec should document that sprint's monitor is coupled to stream-json format.

**Severity**: Low

**Recommendation**: Add a note to v2.08 Section 3.1 under sprint/monitor.py: "Sprint monitor is coupled to stream-json output format (parses NDJSON events for stall detection). This coupling is acceptable because roadmap v1 does not use monitoring. Future roadmap monitoring would need a different approach if text output format is retained."

---

#### IMPACT-5: Sprint test count and modification baseline

**Finding**: v2.08 spec Section 13.6 states "EXISTING: all 14 files unchanged" and lists 14 specific test files. The actual test directory has 14 test files (matching). However, v2.08 also notes "v2.07 modified 9 of the original 14 test files; the regression baseline is the post-v2.07 state." The git status shows all test files as modified on the current branch. The v2.08 spec correctly accounts for this.

**Classification**: NO IMPACT (already accounted for)

---

#### IMPACT-6: Stall watchdog in executor complicates pipeline extraction

**Finding**: The stall watchdog logic (executor.py L126-162) is embedded in the poll loop and uses `config.stall_timeout`, `config.stall_action`, and `monitor.state.stall_seconds`. This logic references sprint-specific config fields and the sprint-specific monitor. It must stay inside `sprint_run_step` when wrapping with `execute_pipeline()`.

**Classification**: REAL IMPACT (reinforces IMPACT-3)

**Why it matters**: The watchdog is not merely a logging concern -- it can terminate the subprocess (`config.stall_action == "kill"`). This side effect must be preserved inside the sprint_run_step callable, not lost during extraction.

**Severity**: Medium

**Recommendation**: Already covered by IMPACT-3 recommendation. Additionally, the v2.08 M2 acceptance criteria should include: "Stall watchdog behavior (--stall-timeout, --stall-action warn|kill) preserved in migrated sprint executor."

### B.2 PERCEIVED IMPACT: Looks Significant But Manageable

#### PERCEIVED-1: v2.05 specified `--output-format text`, actual uses `stream-json`

This initially appears significant because v2.08's pipeline design parameterizes output_format. However, v2.08 already correctly identifies this divergence (D1.4: "default: stream-json for sprint backward-compat; roadmap specifies text"). The spec is aware of the change. The only gap is the implementation detail noted in IMPACT-1.

#### PERCEIVED-2: Two extra source files (debug_logger.py, diagnostics.py)

These files are sprint-specific and will remain in `sprint/`. They do not need to be extracted to `pipeline/`. v2.08's architecture diagram (Section 3.1) does not list them, but this is because they were added after the diagram was drawn, not because they conflict. They should be added to the diagram for completeness but do not affect pipeline extraction.

#### PERCEIVED-3: v2.05 specified ~2,160 lines across 11 files; actual is ~13 files

The line count and file count increased due to v2.07 additions. This does not impact v2.08 because the extraction targets specific modules (process.py, models.py partial), not the entire sprint codebase.

#### PERCEIVED-4: Pre-flight claude binary check in executor

The `shutil.which("claude")` check (executor.py L44-48) is a good practice that roadmap should replicate. It is sprint-specific currently but could be generalized into `pipeline/executor.py`'s pre-flight. Not a spec change -- just a good implementation detail to carry forward.
