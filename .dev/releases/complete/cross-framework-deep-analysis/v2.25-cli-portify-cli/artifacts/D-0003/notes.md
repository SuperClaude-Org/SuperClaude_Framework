## 1. Prompt Splitting Threshold Location

### Decision

**Prompt splitting belongs in the executor, not the prompt builder.**

### Rationale

- Prompt builders (`build_protocol_mapping_prompt()`, `build_analysis_synthesis_prompt()`, etc.) are responsible for content — they assemble the prompt text from config, templates, and embedded file content.
- The executor (`execute_cli_portify()`) controls subprocess invocation and has visibility into the aggregate size of the assembled prompt.
- The 300-line threshold is a structural/routing concern (whether to write to a file vs pass inline), not a content concern. This belongs with the subprocess runner.
- Consistent with cleanup_audit pattern: `build_*_prompt()` returns the prompt string; executor decides how to pass it.

### Implementation Rule

```python
# In executor.py / _run_claude_step():
PROMPT_SPLIT_THRESHOLD_LINES = 300

def _run_claude_step(step: PortifyStep, config: PortifyConfig, ...) -> PortifyStepResult:
    prompt_text = step.prompt
    prompt_lines = prompt_text.count('\n')

    if prompt_lines > PROMPT_SPLIT_THRESHOLD_LINES:
        # Write to portify-prompts.md and pass via --file
        prompts_file = config.work_dir / "portify-prompts.md"
        prompts_file.write_text(prompt_text)
        cmd = ["claude", "--file", str(prompts_file), ...]
    else:
        # Pass prompt inline via stdin or -p flag
        cmd = ["claude", "-p", prompt_text, ...]
```

### Consistency Check

This decision is consistent with:
- Phase 9 (CLI integration) — Phase 9 specifies prompt splitting in `build_release_spec_prompt()` at 50KB for templates, not 300 lines for prompts. These are two separate mechanisms:
  1. **Template file** (`--file` with temp file): release-spec-template.md when content > 50,000 bytes (Phase 6)
  2. **Prompt split** (`portify-prompts.md`): aggregate prompt > 300 lines (executor, all Claude steps)
- Phase 9 explicitly states: "Implement prompt splitting: if aggregate prompt length exceeds 300 lines, split to `portify-prompts.md` (FR-050, AC-010)"
- AC-010: The executor must handle both paths transparently.

## 2. Overwrite Rules for Generated Modules

### Decision

**Only overwrite an existing module if `__init__.py` contains the marker string `"Generated by"` or `"Portified from"`.**

### Confirmed Marker Strings

- `"Generated by"` — standard portification marker
- `"Portified from"` — alternative portification marker

**Both markers** are checked with `in` (substring match), case-sensitive.

### File Checked

`output_dir / "__init__.py"` — the package init file is the canonical marker location.

### Behavior

```python
def _is_portified_module(path: Path) -> bool:
    """Check if existing module was generated by portification."""
    init = path / "__init__.py"
    if init.exists():
        content = init.read_text()
        return "Generated by" in content or "Portified from" in content
    return False
```

If `output_dir.exists()` and `not _is_portified_module(output_dir)`:
→ Raise `PortifyValidationError("NAME_COLLISION", f"CLI module '{cli_name}' already exists")`

If `output_dir.exists()` and `_is_portified_module(output_dir)`:
→ Proceed; overwrite is permitted (previously portified module)

### `__init__.py` Generated Marker

Every generated `__init__.py` must include the marker at the top:
```python
# Generated by: superclaude cli-portify
# Portified from: <workflow_path>
# Generated at: <timestamp>
```

This ensures subsequent runs can detect the module as a portified module and overwrite safely.

## 3. Non-Blocking OQ Documentation

### OQ-007: Agent Discovery Warning Behavior

**Classification**: Non-blocking
**Phase Assignment**: Phase 2 (inventory.py)

**Decision**: Emit `logging.warning()` per unresolved agent file reference. Do not raise an exception or halt the pipeline. Discovery of agent files is best-effort; the inventory is still valid if agent files are absent.

**Downstream impact**: Only Phase 2 inventory.py is affected. Gate G-001 only checks for at least one component with SKILL.md — absent agent files do not cause gate failure.

### OQ-014: Workdir Cleanup Policy

**Classification**: Non-blocking
**Phase Assignment**: Phase 9 (commands.py help text)

**Decision**: No automatic cleanup. The workdir (`.dev/portify-workdir/<cli_name>/`) persists indefinitely after pipeline completion. This enables:
- Resume after interruption
- Artifact inspection post-run
- Debug log review

The user is responsible for cleanup. Document in `--help` output:

```
The work directory at .dev/portify-workdir/<cli-name>/ persists after
completion for artifact inspection and resume support. Clean up manually
when no longer needed.
```
