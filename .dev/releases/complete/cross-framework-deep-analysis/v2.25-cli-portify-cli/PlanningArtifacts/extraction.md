## Functional Requirements

### Phase 0 — Prerequisites

**FR-001**: The system SHALL resolve a `--workflow` argument to a skill directory by globbing `src/superclaude/skills/` and verifying a `SKILL.md` file exists in the resolved path.

**FR-002**: The system SHALL derive a CLI name from the skill directory name by stripping the `sc-` prefix, stripping the `-protocol` suffix, and normalising to kebab-case.

**FR-003**: If `--name` is provided, the system SHALL use that value as the CLI name instead of the derived name.

**FR-004**: The system SHALL detect name collisions against existing `src/superclaude/cli/` modules and raise `PortifyValidationError("NAME_COLLISION", …)` unless the existing module was generated by portification (detected via `"Generated by"` or `"Portified from"` in `__init__.py`).

**FR-005**: The system SHALL verify that the parent directory of the output directory exists and is writable; raise `PortifyValidationError("OUTPUT_NOT_WRITABLE", …)` otherwise.

**FR-006**: The system SHALL create the working directory at `.dev/portify-workdir/<cli_name>/` (parents included) and emit `portify-config.yaml` with resolved paths.

**FR-007**: The system SHALL raise `PortifyValidationError("AMBIGUOUS_PATH", …)` if workflow name resolution matches more than one candidate skill directory.

**FR-008**: The system SHALL raise `PortifyValidationError("INVALID_PATH", …)` if workflow name resolution matches zero candidates.

**FR-009**: The system SHALL raise `PortifyValidationError("DERIVATION_FAILED", …)` if a CLI name cannot be derived from the skill directory name.

**FR-010**: The system SHALL discover and inventory source components including: `SKILL.md`, matching command `.md` files (from `src/superclaude/commands/` and `.claude/commands/sc/`), all `.md` files in `refs/`, all files in `rules/`, `templates/`, `scripts/`, and `decisions.yaml` if present.

**FR-011**: The system SHALL record the line count for each discovered component and emit a `component-inventory.yaml` with `{path, lines, purpose, type}` per component.

**FR-032**: The convergence loop SHALL reach CONVERGED state when zero unaddressed CRITICALs remain and set `status: success`.

**FR-033**: The convergence loop SHALL reach ESCALATED state after 3 iterations are exhausted and set `status: partial`, escalating to the user.

**FR-034**: The convergence loop SHALL set `downstream_ready = true` when `overall >= 7.0`.

**FR-035**: Step `spec-panel-review` SHALL produce an updated `portify-release-spec.md` and a `panel-report.md` with all findings, scores, and convergence status.

## Risk Inventory

**R-001** [HIGH] — Claude subprocess stall: Claude-assisted steps may stall without producing output. The `OutputMonitor` tracks stall seconds but the `stall_action` (kill) implementation is not fully specified; unhandled stalls could block the pipeline indefinitely.  
*Mitigation*: Enforce step timeouts via `timeout` wrapper (exit code 124 → TIMEOUT), implement `stall_action: kill` using `SIGKILL` after `stall_timeout` seconds with `OutputMonitor.growth_rate_bps`.

**R-002** [HIGH] — Turn budget exhaustion: If `max_turns` budget is consumed before high-value steps (10, 11), the pipeline halts without producing a release spec.  
*Mitigation*: `TurnLedger.can_launch()` check; `suggested_resume_budget` computation; `resume_command()` emission. User must re-run with higher budget.

**R-003** [HIGH] — `{{SC_PLACEHOLDER:*}}` sentinels surviving into final spec: If Claude fails to populate all 13 template sections, Gate G-010's `has_zero_placeholders` check will halt but the error may not be self-correcting.  
*Mitigation*: Gate halts immediately; diagnostics report which placeholders remain; user can inspect `portify-release-spec.md` and resume.

**R-004** [HIGH] — Convergence loop ESCALATED without downstream-ready spec: If 3 panel review iterations fail to resolve all CRITICALs, the pipeline emits a partial spec with `downstream_ready = false`.  
*Mitigation*: ESCALATED outcome surfaces to user with `panel-report.md`; user can inspect and manually resolve findings before re-running from step 11.

**R-005** [MEDIUM] — Ambiguous skill name resolution: A fuzzy glob on `src/superclaude/skills/` may match multiple candidates.  
*Mitigation*: `PortifyValidationError("AMBIGUOUS_PATH", …)` with the candidate list displayed; user must provide an explicit path.

**R-006** [MEDIUM] — Name collision with non-portified module: An existing `src/superclaude/cli/<name>/` module not generated by portification would be detected as a collision.  
*Mitigation*: `_is_portified_module()` heuristic; error surfaced early in Step 0 before any work begins.

**R-007** [MEDIUM] — Gate false-positives from line-count minimums: STRICT gates require minimum line counts (50–300) which may reject valid but concise outputs.  
*Mitigation*: Retry limit of 1 for affected steps; Claude prompt instructs comprehensive output; diagnostics report gate failure reason.

**R-008** [MEDIUM] — User review approval file corruption: Manual editing of `phase1-approval.yaml` or `phase2-approval.yaml` with incorrect syntax could cause `check_review_approval()` to fail silently (string match on `"status: approved"`).  
*Mitigation*: Simple string-match is robust to whitespace variance; consider adding YAML schema validation on approval files.

**R-009** [MEDIUM] — `EXIT_RECOMMENDATION` marker absence: If Claude omits the `EXIT_RECOMMENDATION` marker, the step is classified as `PASS_NO_SIGNAL` which propagates through; misclassification could allow a poor output to pass.  
*Mitigation*: Gates G-002, G-003, G-005–G-008, G-010 explicitly check for the marker; PASS_NO_SIGNAL is distinguishable in logs.

**R-010** [MEDIUM] — Agent file discovery gap: The spec notes agent resolution is "best-effort" and not all persona names map to agent files.  
*Mitigation*: Inventory still captures SKILL.md and all other components; missing agents are noted in documentation.

**R-011** [LOW] — Template file size: `release-spec-template.md` may exceed 50 KB, requiring `--file` argument passing. The spec references this as a conditional but the exact threshold handling is not specified in the executor.  
*Mitigation*: `build_release_spec_prompt()` docstring documents the `>50KB` branch; implementation should check file size before embedding inline.

**R-012** [LOW] — `execution-log.jsonl` / `execution-log.md` unbounded growth: Long runs with many retries could produce large log files.  
*Mitigation*: Logs are scoped per `work_dir` run; old work directories are not automatically cleaned. User is responsible for cleanup.

## Open Questions

**OQ-001**: The `TurnLedger` interface and `ledger.can_launch()` semantics are referenced but not defined in the spec. What constitutes "one turn"? Is it per Claude subprocess invocation, per token, or per some other unit?

**OQ-002**: The `OutputMonitor` tracks `stall_seconds` and `growth_rate_bps`, and `stall_action: kill` is declared in `PortifyConfig`, but the kill signal mechanism for the Claude subprocess is not specified. Should the executor use `SIGTERM` followed by `SIGKILL`, or `SIGKILL` directly?

**OQ-003**: The `_build_phase_contracts()` function is referenced in return contract emission but not defined. What is the schema for `phase_contracts`?

**OQ-004**: The `_compute_spec_hash()` function is referenced for `api_snapshot_hash`. What hashing algorithm and content scope should be used (full file, frontmatter only)?

**OQ-005**: The `_count_frs()` function is referenced for `spec_fr_count`. How should functional requirements be counted in the release spec — by parsing a structured section, or by counting FR-NNN patterns?

**OQ-006**: The `_count_source_steps()` function is referenced but not defined. What constitutes a "source step" — the steps in the portification pipeline, or the steps in the portified workflow?

**OQ-007**: Agent file discovery is described as "best-effort" with the note that "not all persona names map to agent files." Should unresolved agent references generate warnings in `result.warnings`, or be silently omitted from the inventory?

**OQ-008**: The template loading branch for files > 50 KB ("loads via `--file` arg") is mentioned in the docstring for `build_release_spec_prompt()` but the exact `--file` mechanism is not specified. Is this a Claude CLI option or a custom file-embedding pattern?

**OQ-009**: The `_get_failure_type()` return contract field is referenced but its enumerated values are not defined. What are the valid failure types (e.g., TIMEOUT, GATE_FAILURE, SIGNAL, BUDGET_EXHAUSTED)?

**OQ-010**: The `_get_resume_substep()` return contract field is referenced. What granularity does "substep" represent — is it the convergence loop iteration number, or a sub-operation within a step?

**OQ-011**: For the `--debug` flag, the spec declares it exists but does not define what additional output or behavior it enables. What logging level or diagnostic information should be shown in debug mode?

**OQ-012**: The spec states that `decisions.yaml` is included in the component inventory, but does not define what `decisions.yaml` contains or how it should be interpreted by downstream Claude-assisted steps.

**OQ-013**: The `PASS_NO_SIGNAL` and `PASS_NO_REPORT` statuses are defined and used in `_determine_status()`, but it is unclear whether these should trigger the `retry_limit=1` retry mechanism or be treated as soft passes. Should they be treated as equivalent to `PASS` for gate purposes?

**OQ-014**: The spec does not define a cleanup/teardown policy for `.dev/portify-workdir/<cli_name>/` after a successful run. Should the workdir be preserved, archived, or deleted?
