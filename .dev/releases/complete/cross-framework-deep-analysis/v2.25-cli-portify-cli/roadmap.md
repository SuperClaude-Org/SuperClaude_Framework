# Project Roadmap: CLI Portify Pipeline (Merged)

## Executive Summary

The CLI Portify system is an enterprise-grade pipeline that converts inference-based SuperClaude skill workflows into deterministic, programmatic CLI runners. It operates as a 12-step, sequentially-gated pipeline orchestrating Claude subprocess invocations with a gate enforcement system, convergence-loop panel review, and full resume/checkpoint support.

**Scope**: 65 requirements (47 functional, 18 non-functional) across 8 technical domains. Complexity score 0.92 (enterprise). The system produces release-quality specifications from skill definitions through automated analysis, specification synthesis, and expert panel review.

**Primary delivery objective**: Deliver `src/superclaude/cli/cli_portify/` and register it in `src/superclaude/cli/main.py` so users can run a supervised, resumable portification workflow with reliable outputs and clear failure handling.

**Key architectural decisions**:
- Sequential-only pipeline execution — simplifies executor, increases wall-clock time
- Domain models extend existing pipeline framework base types — consistent API, tight framework coupling
- 12-gate enforcement with STANDARD/STRICT tiers — high correctness guarantees at implementation cost
- Convergence loop with 3-iteration cap — bounded resource consumption with graceful escalation
- Mandatory Phase 0 gate — resolves contract-affecting open questions before any coding begins
- Observability infrastructure established before Claude-assisted phases — structural enforcement, not advisory

**Critical path**: Framework base types (D-008) must be stable before domain model work begins. The `claude` CLI binary (D-001) is a hard runtime dependency with no fallback. Phase 0 blocks all implementation phases.

**Super-milestone structure for stakeholder reporting**:
- **Milestone A — Foundations**: Phases 0–3 (architecture lock, config/discovery, executor skeleton, gate enforcement)
- **Milestone B — Pipeline Generation**: Phases 4–6 (analysis, specification, synthesis artifacts)
- **Milestone C — Quality Loop and Release Readiness**: Phases 7–10 (panel review, observability completion, CLI integration, verification)

## Risk Assessment

### Highest-Priority Risks

#### Risk 1 — Claude Subprocess Stall

| Attribute | Value |
|-----------|-------|
| Severity | High |
| Likelihood | Medium |
| Impact | Pipeline hang, resource waste, no final artifact |

**Mitigation**: Enforce hard per-step timeouts. Implement stall detection via `OutputMonitor.growth_rate_bps` with SIGTERM grace period then SIGKILL escalation. Step timeouts as hard backstop. Emit diagnostic context on timeout or stall.

**Roadmap response**: Phases 2 (executor timeout classification, kill mechanism) and 8 (observability completion), before relying on long-running synthesis and review steps.

#### Risk 2 — Turn Budget Exhaustion Before Steps 10–11

| Attribute | Value |
|-----------|-------|
| Severity | High |
| Likelihood | Medium |
| Impact | Late-stage halt after substantial cost |

**Mitigation**: Pre-step `TurnLedger.can_launch()` checks. Budget forecast warnings before expensive phases. Accurate `suggested_resume_budget`. Resume command emission in all halted states.

**Roadmap response**: Implement in Phase 2, validate in Phase 10.

#### Risk 3 — Placeholder Leakage into Final Release Spec

| Attribute | Value |
|-----------|-------|
| Severity | High |
| Likelihood | Low–Medium |
| Impact | Invalid downstream artifact; false readiness signal |

**Mitigation**: Explicit placeholder scan. Gate G-010 halts immediately on any `{{SC_PLACEHOLDER:*}}`. Diagnostics report remaining placeholders by name. Retry limit of 1 for Step 10.

**Roadmap response**: Implement in Phases 3 (gate) and 6 (synthesis validation).

#### Risk 4 — Panel Review Escalation Without Downstream-Ready Output

| Attribute | Value |
|-----------|-------|
| Severity | High |
| Likelihood | Medium |
| Impact | Partial output mistaken for completion |

**Mitigation**: Strict state machine separation of `status` from `downstream_ready`. `downstream_ready = false` unless `overall >= 7.0`. Visible escalation details in `panel-report.md`.

**Roadmap response**: Core requirement of Phase 7.

#### Risk 13 — Framework Base Type Stability

| Attribute | Value |
|-----------|-------|
| Severity | High (architectural) |
| Likelihood | Low |
| Impact | All domain models break if `PipelineConfig`, `Step`, `StepResult`, `GateCriteria`, `GateMode`, or `SemanticCheck` API changes |

**Mitigation**: Pin to current framework API. Add integration test that imports all base types. Document exact base type interface contract used.

**Roadmap response**: Confirm in Phase 0 (architecture baseline verification); enforce in Phase 2 (base-type inheritance implementation).

### Medium-Priority Risks

#### Risk 5 — Ambiguous Workflow Resolution

**Mitigation**: Fail early in Phase 1 with candidate list in error message; require explicit path.

**Roadmap response**: Phase 1 validation logic.

#### Risk 6 — Name Collision with Non-Portified Module

**Mitigation**: Marker-based overwrite rule checking `Generated by` / `Portified from` in `__init__.py`; error before any work begins.

**Roadmap response**: Phase 1 collision detection.

#### Risk 7 — Gate False Positives from Line-Count Thresholds

**Mitigation**: Keep thresholds configurable or centralized. Ensure prompts explicitly instruct comprehensive output. Retry limit of 1 provides one recovery cycle.

**Roadmap response**: Phase 3 gate implementation; Phase 4–5 prompt tuning.

#### Risk 8 — Approval File Corruption or Malformed YAML

**Mitigation**: YAML parse + schema validation (not raw string matching on `status: approved`). Silent pass of malformed YAML where field appears in a comment is a safety regression for a resume-critical gate.

**Roadmap response**: Phases 4 and 5 review gate implementations.

#### Risk 9 — Missing EXIT_RECOMMENDATION

**Mitigation**: `PASS_NO_SIGNAL` status is distinguishable from `PASS`. Gates check for marker explicitly. Retry triggers on `PASS_NO_SIGNAL`.

**Roadmap response**: Phase 3 gate logic; Phase 2 status classification.

#### Risk 10 — Agent Discovery Gaps

**Mitigation**: Emit warnings on missing agent files; do not silently suppress. Resolve warning behavior (OQ-007) in Phase 0.

**Roadmap response**: Phase 1 inventory logic.

### Low-Priority Risks

#### Risk 11 — Template Size Handling >50KB

**Mitigation**: Resolve file argument passing mechanism in Phase 0 (OQ-008). Implement in Phase 6.

**Roadmap response**: Phase 0 OQ resolution; Phase 6 synthesis step.

#### Risk 12 — Unbounded Log Growth

**Mitigation**: Keep logs workdir-scoped. Document cleanup expectations (OQ-014, non-blocking from Phase 0).

**Roadmap response**: Phase 8 observability completion.
