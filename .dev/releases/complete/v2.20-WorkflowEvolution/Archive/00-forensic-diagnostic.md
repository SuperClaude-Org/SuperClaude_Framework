# Forensic Diagnostic: Why Bugs Survive the SuperClaude Workflow

**Classification**: Diagnostic only — no fixes proposed
**Date**: 2026-03-08
**Corpus**: 6 specs (v2.03, v2.05, v2.07, v2.08, v2.13, v2.19) + implementation (`src/superclaude/cli/roadmap/`)
**Method**: Cross-spec evidence correlation, spec-to-code comparison, confidence signal audit

---

## Executive Memo

The SuperClaude development pipeline — brainstorm → spec-panel → adversarial → roadmap → tasklist → CLI runner — is a document-processing system that produces documents about code. Every stage validates text. No stage validates behavior.

Three systemic failures allow bugs to survive this pipeline despite its apparent rigor:

**1. Category Error — Documents Are Not Code.** The pipeline checks whether output files have correct frontmatter, sufficient line counts, and proper heading structure. It never checks whether the described code works. A roadmap with perfect formatting and wrong content passes every gate. The pipeline was designed to prevent *bad specifications*; it has no mechanism to prevent *bad implementations*.

**2. Confidence Inflation Through Self-Assessment.** Each stage scores its own output: panel scores of 8.6/10, adversarial convergence of 0.72, tasklist quality of 100%, pass rates of 97.4%. These numbers are generated by the same model that did the work. There is no external calibration. The pipeline's first real-world execution (v2.19) halted immediately — after accumulating five releases of self-assessed high quality.

**3. The Mock Wall.** Every test mocks the Claude subprocess. The test suite validates that orchestration code handles *simulated* success. The first time real Claude output reached the pipeline — 53 bytes of conversational preamble before the `---` delimiter — the gate rejected valid work. End-to-end testing was identified as critical in v2.03, v2.05, v2.07, v2.08, and v2.13. It was deferred in every single release.

The pipeline optimizes for the wrong thing. It produces well-specified work. The problem is that well-specified and well-implemented are different properties, and the pipeline has no way to tell them apart.

## Stage 1: Brainstorm

### What It Claims

Elaborate on objectives, generate ideas, expand scope of consideration.

### What the Evidence Shows

No quality gate exists. Brainstorm output flows directly to spec-panel without validation. The stage produces expansive output where quantity substitutes for depth.

### Exhibits

No brainstorm document in the corpus includes a prioritized risk assessment of LLM output variability — the issue that ultimately broke the pipeline. Brainstorms elaborate on *what to build* without assessing *what can go wrong when an LLM builds it*.

**Vulnerability**: Over-scoping without feasibility analysis. The v2.05 spec targets ~2,160 lines across 11 files with threading, TUI, tmux, and subprocess management — ambitious scope that the brainstorm apparently did not flag as high-risk for agent-generated code.

## Stage 3: Adversarial

### What It Claims

Structured debate challenging the spec via opposing perspectives. Convergence scoring. From the v2.08 merged spec §1:

> **Design principle — no circular self-validation**: The conductor enforces a strict trust boundary. Gate validation (`gate_passed()`) is pure Python — it never invokes Claude to evaluate Claude's output.

### What the Evidence Shows

The adversarial stage debates the *document*, not the *implementation*. It catches logical inconsistencies in text but cannot catch bugs in code that doesn't exist yet. Both debate perspectives share the same model, training data, and context window.

### Exhibits

**Exhibit 3A — What the Adversarial Catches vs What It Misses**

From v2.08 adversarial debates:

| Caught by Adversarial | Missed by Adversarial |
|----------------------|----------------------|
| Spec language allowing ambiguous scope | `_check_frontmatter()` will fail on LLM preamble |
| Missing deferred-feature tracking | `--verbose` flag injects diagnostics into stdout |
| Architectural coupling concerns | Semantic checks replicate the pipeline gate bug |
| Process workflow gaps | Prompts don't specify output length expectations |
| Risk assessment incompleteness | `proc._process` private attribute access in executor |

The left column is *textual* analysis. The right column is *implementation* analysis. The adversarial stage can only perform the left.

**Exhibit 3B — The 10-15% That Drops Out**

From v2.07 retrospective §4.5:
> ~85-90% of adversarial conclusions were incorporated. Notable omissions:
> - Strategy 2's 6-field structured error format was simplified to 2-field
> - Three v1.1 forward/deferral notes were not documented
> - Strategy 3's fourth standalone criterion ("session-start executable") was dropped

The adversarial stage caught these issues. The implementation dropped them anyway. The pipeline has no mechanism to verify that adversarial findings were actually incorporated — the check happens at the text level and the loss happens at the implementation level.

**Exhibit 3C — Convergence Is Self-Agreement**

From v2.13:
> Adversarial debate (convergence 0.72) settled that executor unification is premature

Convergence 0.72 means the model's two roleplay perspectives reached 72% agreement. This is the model agreeing with itself through two lenses. A genuine adversarial test would require one perspective to have information the other lacks. Both perspectives share the same training data, same context, same model weights.

## Stage 4: Roadmap

### What It Claims

Phased implementation plan with milestones, timelines, and dependencies derived from the approved spec.

### What the Evidence Shows

No quality gate on roadmap output. Timeline estimates lack empirical basis. Dependency chains describe *intended* order, not *proven* requirements.

### Exhibits

**Exhibit 4A — Ungrounded Timeline Formulas**

From v2.05 (sprint spec NFR-003):
> Phase timeout: `max_turns * 120 + 300` seconds.

120 seconds per turn is a guess. The v2.07 retrospective shows actual throughput varies: Phase 1 at 2.59 tasks/min vs Phase 4 at 1.25 tasks/min. The formula was never empirically validated.

From v2.19 (reliability spec, Appendix A):
> WS-1: Gate tolerance: 2–3h; WS-4: Schema parity: 4–6h; Total: 10.5–16.5h

These estimates come from the spec-writing agent, not from prior execution data. The range (10.5–16.5h) suggests uncertainty, but the roadmap treats the midpoint as a plan.

**Exhibit 4B — Dependencies That Assume Previous Phase Output Format**

From the v2.08 merged spec, the 8-step pipeline:
```
extract → generate-A/B (parallel) → diff → debate → score → merge → test-strategy
```

Each step's prompt says "Read the provided [previous artifact]." But what format will the previous artifact be in? The extract step's prompt (from `roadmap/prompts.py`) says:

> "Read the provided specification file and produce a requirements extraction document."

This produces *whatever Claude decides extraction looks like*. The diff step then reads two roadmap variants — whatever Claude decided roadmaps look like. Each step assumes the previous step's output matches an implicit schema that is never formally specified or validated beyond frontmatter field presence.

## Stage 5: Tasklist

### What It Claims

Numbered, actionable tasks with acceptance criteria, dependencies, and tier classification. Quality measured across 8 dimensions.

### What the Evidence Shows

Quality dimensions measure *description format*, not *execution feasibility*. Tasks that read well may lose critical spec context during the handoff to the CLI runner.

### Exhibits

**Exhibit 5A — Quality Dimensions That Don't Predict Execution Success**

From v2.07 retrospective §3.2:
> All 39 tasks scored 100% across 8 quality dimensions (ID format, metadata, standalone descriptions, step labels, acceptance criteria, dependencies, rollback, tier distribution). Claude Code executed them without confusion or backtracking.

The 8 dimensions: ID format, metadata, standalone descriptions, step labels, acceptance criteria, dependencies, rollback instructions, tier distribution. Every dimension assesses the *text* of the task description. No dimension assesses whether executing the task will produce correct code.

**Exhibit 5B — Acceptance Criteria Are Structural, Not Behavioral**

A typical task acceptance criterion:
```
AC: Function locates YAML frontmatter using regex pattern
AC: Required fields are validated
AC: Backward compatibility with existing files maintained
```

"Function locates YAML frontmatter" — any regex satisfies this. Claude can write a regex that locates frontmatter in some cases and fails in others, and the AC is met. "Required fields are validated" — presence checking satisfies this, even if the validation is trivially permissive. The ACs describe *what* should exist, not *how well* it should work.

**Exhibit 5C — Spec Context Loss During Handoff**

Task descriptions reference spec sections (e.g., "per spec section 4.1.3") but the CLI runner doesn't embed the referenced section in the prompt. The runner provides:
- The task description (a summary)
- File inputs (outputs of previous pipeline steps)

It does NOT provide:
- The full spec section being referenced
- The reference implementation from the spec
- The edge cases the spec identifies
- The test cases the spec requires

The agent executing the task must reconstruct or guess the spec's intent from a summary.

## Stage 6: CLI Runner (Gates & Execution)

### What It Claims

Automated step execution with file-on-disk gates between steps. From the v2.08 merged spec §1:

> Gate validation (`gate_passed()`) is pure Python — it never invokes Claude to evaluate Claude's output. This eliminates the circular self-validation failure mode.

### What the Evidence Shows

Gates validate document structure. They check whether output files *look correct*, not whether they *are correct*. The test suite validates gates against pre-written mock files, never against real Claude subprocess output.

### Exhibits

**Exhibit 6A — The Gate Function (What It Actually Checks)**

From `pipeline/gates.py`:
```python
def gate_passed(output_file: Path, criteria: GateCriteria) -> tuple[bool, str | None]:
    # EXEMPT: always passes
    if tier == "EXEMPT":
        return True, None
    # File must exist
    if not output_file.exists():
        return False, f"File not found: {output_file}"
    # File must be non-empty
    content = output_file.read_text(encoding="utf-8")
    if len(content.strip()) == 0:
        return False, f"File empty (0 bytes): {output_file}"
    # STANDARD: minimum line count
    lines = content.splitlines()
    if len(lines) < criteria.min_lines:
        return False, f"Below minimum line count: {len(lines)} < {criteria.min_lines}"
    # STANDARD: YAML frontmatter fields
    # STRICT: semantic checks (heading gaps, duplicate headings, convergence score range)
```

What this checks: file exists, non-empty, line count, frontmatter field names, heading structure.
What this doesn't check: content correctness, requirement coverage, algorithmic validity, behavioral accuracy.

A roadmap with 150 lines, correct frontmatter, and proper heading hierarchy — but containing a completely infeasible implementation plan — passes the `MERGE_GATE`.

**Exhibit 6B — The Brittle Frontmatter Check (The Actual Crash Site)**

From `pipeline/gates.py`, the function that halted the pipeline on first real execution:
```python
def _check_frontmatter(content, required_fields, output_file):
    stripped = content.lstrip()
    if not stripped.startswith("---"):
        return False, f"YAML frontmatter missing or unparseable in {output_file}: no opening ---"
```

From v2.19 §1:
> The extraction output is **valid** — 190 lines, all required frontmatter fields present, correct complexity analysis. The pipeline rejects correct work because 53 bytes of LLM conversational preamble ("Now I have the full spec. Let me produce the extraction document.") precede the `---` delimiter.

**Exhibit 6C — The Bug Replicated in a Different Layer**

From `roadmap/gates.py`, TWO semantic check functions independently replicate the same brittle parsing:

```python
def _frontmatter_values_non_empty(content: str) -> bool:
    stripped = content.lstrip()
    if not stripped.startswith("---"):
        return False                          # <-- same byte-position assumption

def _convergence_score_valid(content: str) -> bool:
    stripped = content.lstrip()
    if not stripped.startswith("---"):
        return False                          # <-- same byte-position assumption
```

Fixing `pipeline/gates.py` (WS-1 in the v2.19 spec) would allow the gate to pass — but the semantic checks would still fail on preamble. The bug exists at two layers. The fix targets one layer.

**Exhibit 6D — The `--verbose` Flag Nobody Tested**

From v2.19 §1.3:
> The `ClaudeProcess.build_command()` includes `--verbose` in every subprocess invocation. It is **unknown** whether `--verbose` with `output_format="text"` injects diagnostic text into stdout, contributing to or causing the preamble.

The pipeline was developed, tested (with mocks), reviewed (by simulated experts), adversarially debated, and released — without anyone checking whether the subprocess invocation flags produce parseable output.

**Exhibit 6E — Prompts That Don't Align With Gates**

From `roadmap/prompts.py` (extract prompt):
> "Your output MUST begin with YAML frontmatter delimited by --- lines containing:
> - functional_requirements: (integer count of identified requirements)
> - complexity_score: (float 0.0-1.0 assessing overall complexity)"

From `roadmap/gates.py` (extract gate):
```python
EXTRACT_GATE = GateCriteria(
    required_frontmatter_fields=["functional_requirements", "complexity_score", "complexity_class"],
    min_lines=50,
)
```

The prompt tells Claude to produce a `complexity_score` as a "float 0.0-1.0." The gate checks only that a field named `complexity_score` *exists*. Claude could produce `complexity_score: "high"` and pass the gate. The prompt requests format; the gate validates presence.

Further: the prompt says nothing about output length. The gate requires `min_lines: 50`. If Claude produces a concise 30-line extraction with perfect content, the gate rejects it for being too short. The prompt and the gate are enforcing different contracts, and neither knows about the other's requirements.

**Exhibit 6F — Schema Drift Between Protocol and CLI**

From v2.19 §1.2:
> The source protocol specifies **17+ frontmatter fields** including `spec_source`, `generated`, `generator`, `nonfunctional_requirements`, `total_requirements`, `domains_detected`, `risks_identified`, `dependencies_identified`, `success_criteria_count`, `extraction_mode`, and a nested `pipeline_diagnostics` block.
>
> The current CLI extract prompt requests only 3 fields (`functional_requirements`, `complexity_score`, `complexity_class`) and the gate validates only those 3.

The CLI was ported from the protocol spec. The protocol evolved. The CLI gates didn't. There is no mechanism anywhere in the workflow to detect this drift.

**Exhibit 6G — Implementation Divergences From Spec**

From `roadmap/executor.py`:
```python
# Private attribute access — spec defines start()/wait()/terminate() as public API
while proc._process is not None and proc._process.poll() is None:
```

```python
# Defensive fallback not in spec — spec says "exactly 2 agents"
agent_b = config.agents[1] if len(config.agents) > 1 else config.agents[0]
```

```python
# Dead import — module not in any spec
from ..pipeline.deliverables import decompose_deliverables
from ..pipeline.models import Deliverable, ...
```

These patterns suggest the code was generated from a prompt summary of the spec rather than with full spec context. The code handles cases the spec says won't occur, accesses internals the spec doesn't expose, and imports modules the spec doesn't define.

## Stage 7: Post-Execution (The Missing Stage)

### What It Claims

Nothing. This stage doesn't exist.

### What the Evidence Shows

There is no automated post-execution validation anywhere in the pipeline. After the CLI runner reports "8/8 PASS," no stage:
- Runs the generated code
- Executes the test strategy against real outputs
- Compares deliverables to spec requirements
- Validates that the implementation matches behavioral expectations

The first validation of actual behavior is when a user runs the deliverable.

## Cross-Cutting Failure Patterns

### Pattern 1: Proxy Stacking

Each stage produces a quality signal that is a *proxy* for the real thing:

```
Panel score (8.6)  →  "spec is good"           →  proxy for: spec will produce working code
Convergence (0.72) →  "debate was productive"   →  proxy for: design is robust
Quality (100%)     →  "tasks are well-formed"   →  proxy for: tasks will execute correctly
Gate (8/8 PASS)    →  "output looks right"      →  proxy for: output is right
Pass rate (97.4%)  →  "tasks were completed"    →  proxy for: feature works
Grade (B+)         →  "release succeeded"       →  proxy for: production-ready
```

Each stage treats the previous stage's proxy as the real thing. The chain compounds: 8.6 → 0.72 → 100% → 8/8 → 97.4% → B+. Each number is locally valid (the measurement is correct for what it measures). The chain is globally invalid (no number measures whether the deliverable works).

### Pattern 2: Spec-Compliance Quality Standard

Quality is measured against the spec, not against production behavior:

From v2.07 §9:
> "**The work was done completely and accurately according to the spec**, with one known exception."

The v2.07 sprint found 7 telemetry bugs and had no E2E test. But because the *spec* didn't require E2E testing, and the *spec's* criteria were met, the grade is B+. The quality standard allows every release to claim success while accumulating production-impacting bugs — because the bugs are outside the spec's definition of quality.

### Pattern 3: Architectural Accretion Without Coverage Growth

Each release adds architectural sophistication. v2.08 adds `pipeline/` shared module, `PipelineConfig`, `Step`, `GateCriteria`, `SemanticCheck`. v2.13 adds lifecycle hooks, characterization tests, file-embedding logic. The codebase becomes more abstract and more layered.

But test coverage stays flat. v2.13 reports ~45% sprint executor coverage and targets ≥70%. Roadmap coverage is unspecified. Each new abstraction layer is a new surface for bugs, with the same testing infrastructure (mocks of expected behavior) that didn't catch the bugs in the simpler version.

### Pattern 4: The Retrospective-Deferral Cycle

```
Release N: Ship feature
Release N retrospective: "We found bugs X, Y, Z. E2E testing is needed."
Release N+1 scope: "Bug fixes from N are out of scope. E2E testing deferred."
Release N+1 retrospective: "We found bugs X, Y, Z (still), plus A, B, C."
Release N+2 scope: "Bug fixes from N+1 are out of scope..."
```

The retrospective documents the right findings. The next release defers them. The cycle repeats. The retrospective is a *documentation* exercise, not a *change* exercise.
