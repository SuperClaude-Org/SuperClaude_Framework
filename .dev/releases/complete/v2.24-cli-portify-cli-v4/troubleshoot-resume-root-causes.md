# Root Cause Analysis: --resume Triggering Roadmap Regeneration
# v2.24-cli-portify-cli-v4

**Date**: 2026-03-13
**Reported behavior**: `superclaude roadmap run --resume` regenerated new roadmaps instead of
skipping to the failing `spec-fidelity` phase.
**Sources**: Agent 1 (artifact analysis) + Agent 2 (resume logic analysis)

---

## Context

At the time of the reported `--resume` invocation:
- All 8 pipeline steps (extract → test-strategy) had PASS status in `.roadmap-state.json`
- `spec-fidelity` had FAIL status (attempt 2), `fidelity_status: "fail"`
- The spec file `portify-release-spec.md` had been modified to fix the detected spec variation
  before `--resume` was run

The state file `spec_hash`:
```
3d1a6d1581bd8352ace0f80faf845b1131b8df36b3fe7e458e299a7647dfbaa9
```

---

## How --resume Works (Architecture Summary)

`_apply_resume()` (`executor.py:1055-1131`) does NOT consult the `steps` dictionary in
`.roadmap-state.json` to decide what to skip. It exclusively re-evaluates each step's gate
against on-disk output files. First-failure-cascades: the moment any gate fails, that step
and ALL subsequent steps are included for re-run, unconditionally.

The state file is used for ONLY two things during resume:
1. `spec_hash` comparison (stale-spec detection)
2. Post-pipeline `validation` key (skip already-completed validation)

---

## Root Cause 1: Spec File Modified Before --resume — SHA-256 Hash Mismatch Triggers Full Pipeline Re-run

**Confidence: HIGH — this is the most likely cause of the reported behavior**

**Mechanism** (`executor.py:1067-1079`):
```python
saved_hash = state.get("spec_hash", "")
current_hash = hashlib.sha256(config.spec_file.read_bytes()).hexdigest()
if saved_hash and saved_hash != current_hash:
    print("WARNING: spec-file has changed since last run. Forcing re-run of extract step.")
    force_extract = True
```

When `force_extract = True`, the `extract` step is forcibly included in the run list
(`executor.py:1111-1115`). Because of the cascade rule (`found_failure = True` set on extract),
every downstream step is also included — including the `generate-opus-architect` and
`generate-haiku-architect` parallel group that produces the roadmap files.

**Why this matches the reported behavior**: The user states that `--resume` was run "due to a
spec variation that was detected and fixed." Fixing the spec variation required editing
`portify-release-spec.md`. Even a single character change to the spec file causes the stored
SHA-256 hash to mismatch the recomputed hash, triggering a full pipeline re-run starting from
`extract`. This regenerates `extraction.md` → both roadmap files → `diff-analysis.md` →
`debate-transcript.md` → `base-selection.md` → `roadmap.md` → `test-strategy.md` → and
finally a new `spec-fidelity.md` — which then fails spec-fidelity again because the spec
issues (DEV-002, DEV-003) are in the roadmap content, not in the spec file itself.

**The warning printed to stderr** (`"WARNING: spec-file has changed since last run..."`) may
not have been visible to the user if the output was not captured or scrolled past.

## Root Cause 2: gate_passed() Re-evaluates STRICT Semantic Checks on Existing spec-fidelity.md — Gate Failure Cascades

**Confidence: MEDIUM — secondary cause, explains why spec-fidelity re-ran in any scenario**

**Mechanism**:

Even if the spec file hash matched (no change), `_apply_resume()` re-evaluates
`SPEC_FIDELITY_GATE` against the existing `spec-fidelity.md`. The existing file contains:

```yaml
high_severity_count: 3
tasklist_ready: false
```

`SPEC_FIDELITY_GATE` (`gates.py`) is STRICT and includes semantic check
`high_severity_count_zero`, which requires `high_severity_count == 0`. This check FAILS the
existing on-disk file. Therefore `spec-fidelity` is correctly flagged as needing re-run.

**The problem**: re-running `spec-fidelity` in isolation is only possible if no earlier step
was also re-run. If Root Cause 1 triggered (spec file changed), all earlier steps including
`generate-*` already cascaded, so both roadmap files were regenerated before `spec-fidelity`
ran. The new roadmaps, being freshly generated by Claude from the modified spec, still
contained the same logical gaps (DEV-002, DEV-003 are roadmap content issues, not spec issues)
— so the new `spec-fidelity` run also found `high_severity_count > 0` and failed again.

**Key insight**: The user expected `--resume` to run ONLY `spec-fidelity`. It did not, because
the spec edit triggered the cascade before `spec-fidelity` was even evaluated.

## Root Cause 3: _apply_resume() Does Not Read the steps Dictionary from State — Recorded PASS Status Cannot Protect Steps from Cascade

**Confidence: MEDIUM — architectural gap that makes the system fragile to any upstream invalidation**

**Mechanism** (`executor.py:1055-1131`):

The state file contains a `steps` dictionary with explicit PASS/FAIL per step:
```json
"extract": {"status": "PASS", "attempt": 1},
"generate-opus-architect": {"status": "PASS", "attempt": 1},
...
```

`_apply_resume()` reads this data structure but uses NONE of it for skip/run decisions.
Instead it calls `gate_fn(step.output_file, step.gate)` for each step from scratch.

**Why this matters**: There is no mechanism to say "this step is known-PASS from the previous
run, protect it from cascade." If anything upstream of a step causes `found_failure = True`
(a changed spec, a deleted file, a tightened gate criterion), that step is re-run regardless
of its recorded PASS status.

**Consequence for the reported behavior**: Even if the spec file had NOT changed, the cascade
rule means that a single failing gate at any step forces all downstream steps to re-run. In
the reported scenario, the cascade was triggered at `extract` (Root Cause 1). With the state
dictionary ignored, there is no fallback protection to say "we know generate-* passed, don't
regenerate the roadmaps."

**Design trade-off being exposed**: Gate re-evaluation is intentional (ensures file content is
valid, not just that a previous run claimed it passed). But combined with the cascade rule and
spec-hash sensitivity, it creates a sharp cliff: a spec file change of any size resets the
entire pipeline, even when only the validation phase needs to re-run.

## Correlation Summary

| # | Root Cause | Trigger | Effect | Evidence |
|---|-----------|---------|--------|----------|
| 1 | Spec file edited before `--resume` | SHA-256 mismatch → `force_extract = True` | Full cascade from `extract`, regenerates both roadmap files | `executor.py:1067-1079`, user stated spec was "fixed" |
| 2 | SPEC_FIDELITY_GATE STRICT semantic fail on existing file | `high_severity_count: 3` in on-disk `spec-fidelity.md` | `spec-fidelity` always re-runs; new roadmaps still fail for same reasons | `gates.py` semantic checks, `spec-fidelity.md` frontmatter |
| 3 | State file `steps` PASS records not consulted | `_apply_resume()` only uses `spec_hash` from state | No protection against cascade; recorded PASS cannot shield earlier steps | `executor.py:1055-1131` — state `steps` dict read but never used |

**Primary causal chain**:
```
spec-file edited to "fix variation"
  → SHA-256 hash mismatch detected in _apply_resume()
    → force_extract = True
      → extract step cascade: found_failure = True
        → ALL downstream steps included (generate-opus, generate-haiku, diff, debate, score, merge, test-strategy, spec-fidelity)
          → New roadmaps generated by Claude
            → spec-fidelity re-runs on new roadmaps
              → Still finds high_severity_count > 0 (DEV-002/DEV-003 are roadmap content gaps, not spec gaps)
                → spec-fidelity FAIL again
```

## Recommended Fixes (for future consideration)

1. **Separate spec-patch from spec-change**: Introduce a `--accept-spec-change` flag or a
   spec-hash update command that updates the stored hash without triggering a full pipeline
   re-run, for cases where the spec edit was a documentation fix (e.g., formalizing an
   accepted deviation) rather than a functional change.

2. **Honor recorded PASS status as a cascade guard**: When `force_extract` is triggered by a
   spec change, allow the user to pass `--preserve-outputs` to skip regenerating steps whose
   state-file status is PASS and whose output files still exist. Useful when the spec edit
   was a backwards-compatible documentation update.

3. **Warn loudly before full cascade**: When a spec-hash mismatch would trigger a full
   cascade (28+ minute re-run), print an interactive prompt (or `--yes` flag) requiring
   explicit confirmation before proceeding, so the user understands the scope of what is
   about to re-run.
